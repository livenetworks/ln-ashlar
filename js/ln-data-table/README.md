# ln-data-table

> Event-driven data table — you describe the columns and the row template in HTML, the component handles sort, filter, search, selection, virtual scroll and keyboard navigation. You feed it data via events.

## Philosophy

`ln-data-table` is the answer to "I have a list of records and I want a real
table around them." It deliberately does not fetch, paginate, or own a data
source. Instead, it announces what it needs (`ln-data-table:request-data`)
and waits for someone to hand it data back (`ln-data-table:set-data`). That
"someone" is a tiny coordinator script you write per page — typically 10-20
lines that bridges the table and your data source (an `ln-store` IndexedDB
cache, a `fetch()` to a Laravel endpoint, or a static JS array).

This split exists for one reason: the table is the same regardless of where
the data comes from. Server-paginated, fully-cached, pre-loaded into the
page — the table doesn't care. Sort, filter and search produce a single
`request-data` event with the current intent; you decide whether to satisfy
it from memory or from a server round-trip. The component pays attention to
markup-driven state — `data-ln-col`, `data-ln-row`, `data-ln-cell` — and
ignores everything else, which means consuming projects can wrap the table
in any layout, header, toolbar or footer without breaking the rendering
contract.

The shape of the markup is also a deliberate choice. The table is a real
`<table>` (not a `<div>`-grid), every column header is a `<th>` with a
`data-ln-col="field"` mapping, and every row is cloned from a `<template>`
sitting next to the table — not generated by JS string concatenation. This
keeps the HTML readable in DevTools, lets you change the cell layout without
touching JS, and makes it impossible to inject untrusted strings as markup.
Sort indicators are three sibling `<svg>` icons with `data-ln-sort-icon`
attributes — CSS swaps which one shows based on a class on the `<th>`.
Nothing about the row template, the sort icons, or the empty state is
JavaScript-managed visual state; it's all CSS reacting to attributes the
component sets.

Trade-offs the design accepted:
- **No built-in pagination.** Pagination forces a 2-D mental model
  (page + sort + filter); virtual scroll keeps it 1-D. The data set is
  treated as a list — the coordinator is free to fetch chunks, but
  the table itself shows one list.
- **No built-in fetch / no built-in data source.** The component would
  need a hundred config options to be flexible; the coordinator pattern
  keeps the contract tiny.
- **One row template per table.** If you need fundamentally different
  row layouts, those are different tables. (Different statuses sharing
  one layout but different cell content is fine — that's what
  `data-ln-cell-attr` is for.)

## Markup anatomy

Below is a complete minimal table. Each piece exists for a reason — read
the prose after the snippet for the *why*.

```html
<section data-ln-data-table="documents" data-ln-data-table-selectable id="documents-table">

	<header>
		<h3>Documents</h3>
		<aside>
			<label>
				<svg class="ln-icon ln-icon--sm" aria-hidden="true"><use href="#ln-search"></use></svg>
				<input type="search" placeholder="Search..." data-ln-data-table-search>
			</label>
		</aside>
	</header>

	<div class="table-container">
		<table>
			<thead>
				<tr>
					<th data-ln-col-select></th>
					<th data-ln-col="title">
						Title
						<button data-ln-col-sort aria-label="Sort">
							<svg class="ln-icon" aria-hidden="true" data-ln-sort-icon="none"><use href="#ln-arrows-sort"></use></svg>
							<svg class="ln-icon" aria-hidden="true" data-ln-sort-icon="asc"><use href="#ln-arrow-up"></use></svg>
							<svg class="ln-icon" aria-hidden="true" data-ln-sort-icon="desc"><use href="#ln-arrow-down"></use></svg>
						</button>
					</th>
					<th data-ln-col="department">
						Department
						<button data-ln-col-filter aria-label="Filter">
							<svg class="ln-icon" aria-hidden="true"><use href="#ln-filter"></use></svg>
						</button>
					</th>
					<th>Actions</th>
				</tr>
			</thead>
			<tbody data-ln-data-table-body></tbody>
			<tfoot>
				<tr>
					<td colspan="99">
						<span data-ln-data-table-total></span> items
						<span data-ln-data-table-filtered-wrap> · <span data-ln-data-table-filtered></span> filtered</span>
						<span data-ln-data-table-selected-wrap> · <span data-ln-data-table-selected></span> selected</span>
					</td>
				</tr>
			</tfoot>
		</table>
	</div>

	<template data-ln-template="documents-row">
		<tr data-ln-row>
			<td><input type="checkbox" data-ln-row-select></td>
			<td data-ln-cell="title"></td>
			<td data-ln-cell="department"></td>
			<td>
				<button data-ln-row-action="edit" aria-label="Edit"><svg class="ln-icon" aria-hidden="true"><use href="#ln-edit"></use></svg></button>
				<button data-ln-row-action="delete" aria-label="Delete"><svg class="ln-icon" aria-hidden="true"><use href="#ln-trash"></use></svg></button>
			</td>
		</tr>
	</template>

	<template data-ln-template="documents-empty">
		<tr><td colspan="99"><h3>No documents yet</h3></td></tr>
	</template>

	<template data-ln-template="documents-empty-filtered">
		<tr><td colspan="99">
			<h3>No results</h3>
			<button data-ln-data-table-clear-all>Clear all filters</button>
		</td></tr>
	</template>

	<template data-ln-template="column-filter">
		<div class="column-filter-dropdown">
			<input type="search" data-ln-filter-search placeholder="Search...">
			<ul data-ln-filter-options></ul>
			<button data-ln-filter-clear>Clear filter</button>
		</div>
	</template>
</section>
```

### The component root — `<section data-ln-data-table="...">`

The root carries `data-ln-data-table="documents"`, where `"documents"` is
the **table name** — a unique identifier per page. The name is echoed in
every event the table dispatches (`detail.table === 'documents'`), so a
coordinator listening at `document` level can target one table among many.
The element is a `<section>` because the contents form a self-contained
piece of UI with a heading. It could equally be an `<article>` — what
matters is that it's a real semantic container, not a `<div>`.

`data-ln-data-table-selectable` is opt-in. Adding it switches on the row
checkbox + select-all behavior and emits `ln-data-table:select` events.
Leave it off if rows are not multi-selectable.

### The toolbar — `<header>` with search

The component does not insert the toolbar; you build it. The component
only watches for one element inside it: an `<input>` (or any element)
carrying `data-ln-data-table-search`. Every keystroke updates
`currentSearch`, dispatches `ln-data-table:search`, and triggers a fresh
`request-data`. There's no debounce — if the data is in memory the cost
is trivial; if it goes to a server, the coordinator is the right layer
to debounce.

### The table — `<table>`, not a wrapping `<div>`

`<table>` is the root the component operates on. The `<table>` belongs
inside an outer `<div class="table-container">` so horizontal overflow
on narrow screens scrolls the table without breaking the page chrome.
The component does not require the container — it requires the
`<table>`.

### Column headers — `<th data-ln-col="field">`

Each header `<th>` is bound to a data field via `data-ln-col="field"`.
The string `field` is the property name read from records in the row
template — `<th data-ln-col="title">` matches `<td data-ln-cell="title">`.
Without `data-ln-col`, the column has no field to sort or filter on; it's
treated as a presentational column (e.g. the "Actions" header in the
example above is plain `<th>Actions</th>` because there is no underlying
data field).

### The sort button — three icons, one source of truth

The sort button contains **three** `<svg>` siblings with distinct
`data-ln-sort-icon` values: `"none"`, `"asc"`, `"desc"`. All three are in
the DOM at all times. CSS shows exactly one based on a class on the
ancestor `<th>`:

- Default (no class on `<th>`) → `"none"` icon visible (`ln-arrows-sort`)
- `<th class="ln-sort-asc">` → `"asc"` icon visible (`ln-arrow-up`)
- `<th class="ln-sort-desc">` → `"desc"` icon visible (`ln-arrow-down`)

The component never swaps an `<svg>`, never replaces an icon URL, and
never sets inline styles. It toggles a class on the `<th>`. CSS does
the rest. This is the cleanest example in the library of *markup-driven
state*: every visual change is a class flip, and the icons exist
declaratively in HTML where you can read them.

The sort cycle is three states: click an inactive column → `asc`,
click again → `desc`, click again → cleared (no sort). Each click
emits `ln-data-table:sort` and triggers `request-data`.

### The filter button — `<button data-ln-col-filter>`

A header may contain a filter button in addition to (or instead of) a
sort button. Clicking it clones the `column-filter` template, populates
it with the unique values for that column, and appends it inside the
`<th>`. CSS positions it (the `<th>` is `position: relative`, the
dropdown is `position: absolute`). When at least one value is unchecked,
JS adds `ln-filter-active` to the button — CSS draws an active dot.

### The row template — `<template data-ln-template="{name}-row">`

The row template lives **inside the component root**, named
`{table}-row`. The component clones it once per record and fills the
cells. Two attributes do the filling:

- `data-ln-cell="field"` → `el.textContent = record[field]`
- `data-ln-cell-attr="field:attr"` → `el.setAttribute(attr, record[field])`,
  comma-separated for multiple: `"id:data-id,url:href"`

Why a `<template>` and not a static `<tr>`? A static `<tr>` would render
empty in the initial paint and the user would see a broken row. A
`<template>` is inert until cloned — the component owns when rows
appear. Why not build them in JS? Because then the row layout lives in
your script file and you have to redeploy JS to add a column. With a
template, the markup stays in HTML where designers and humans can read
it.

`data-ln-row` on the `<tr>` marks it as the click target. Clicking
the row emits `ln-data-table:row-click` (with the full record on
`detail.record`). Clicking a button inside the row, a checkbox, or a
link short-circuits this — those have their own handlers.

`data-ln-row-action="edit"` / `"delete"` on a `<button>` inside the row
emits `ln-data-table:row-action` with the action name. The component
does not perform the action; it announces it. The coordinator decides
what "edit" means.

### Empty states — two templates, two situations

`{table}-empty` shows when the table has zero records total
(`detail.total === 0`). `{table}-empty-filtered` shows when total is
non-zero but filters reduced the visible set to zero. They are different
templates because the messaging is different — "you haven't created
anything yet, here's a CTA" versus "your filters returned no results,
here's a clear-filters button." The empty-filtered template typically
contains a `<button data-ln-data-table-clear-all>` which clears all
column filters and re-requests data.

### The column-filter dropdown template

`{table}-column-filter` (or the shared default `column-filter`) is the
template cloned every time a filter button is clicked. The dropdown
renders unique column values as checkboxes. A search input inside the
dropdown lets the user filter the option list itself (auto-hidden when
there are 8 or fewer values). A `[data-ln-filter-clear]` button clears
just that column's filter.

### The footer — `data-ln-data-table-total` / `-filtered` / `-selected`

The footer is yours. The component looks for three optional spans:

- `data-ln-data-table-total` — gets the total count, formatted via
  `Intl.NumberFormat` for the document's `lang`
- `data-ln-data-table-filtered` — gets the filtered count when filters
  reduce the visible set; the wrapper carrying
  `data-ln-data-table-filtered-wrap` is hidden via the `hidden` class
  when no filter is active
- `data-ln-data-table-selected` — gets the selection count; same
  wrapper-hide pattern via `data-ln-data-table-selected-wrap`

The wrap pattern lets you write punctuation that disappears with the
value: `<span data-ln-data-table-filtered-wrap> · <span data-ln-data-table-filtered></span> filtered</span>` shows
" · 12 filtered" only when filtering is active, and disappears entirely
otherwise — no awkward " · 0 filtered" or empty leading separator.

## States & visual feedback

Everything visual reacts to JS-toggled classes or attributes — never to
inline styles set by JS. The component's job is to flip the right
attribute or class; CSS owns how that looks.

| Trigger | What JS sets | What CSS does |
|---|---|---|
| Sort cycle | Adds `ln-sort-asc` or `ln-sort-desc` to `<th>` | Hides the `data-ln-sort-icon="none"` SVG, shows the matching directional one |
| Active filter | Adds `ln-filter-active` to filter `<button>` | Draws an active dot via `::after` |
| Row hover (selectable or clickable) | (no JS — pure CSS via `[data-ln-row]:hover`) | Background shifts to `--bg-sunken` |
| Row selected | Adds `ln-row-selected` to `<tr>` | Background tint via `--color-accent-tint` |
| Row keyboard focus | Adds `ln-row-focused` to `<tr>` | Outline via `--color-accent-focus` |
| Loading | Adds `ln-data-table--loading` to root | Dims `<tbody>` and disables pointer events |
| Empty state | Renders `{table}-empty` or `{table}-empty-filtered` template into `<tbody>` | Template content owns its own visual |
| Filter active total | Adds/removes `hidden` on `[data-ln-data-table-filtered-wrap]` | Wrapper disappears when count == total |
| Selection active | Adds/removes `hidden` on `[data-ln-data-table-selected-wrap]` | Wrapper disappears when count == 0 |
| Virtual scroll | Inserts `<tr class="ln-data-table__spacer">` rows | Spacer rows render with zero padding/border, just height |

Read it as: JS toggles a class, CSS shows the result. There are no
JS-driven inline styles in the rendering path.

## Attributes

### Container — on `<section data-ln-data-table>` and direct children

| Attribute | On | Description | Why it exists |
|---|---|---|---|
| `data-ln-data-table` | wrapper | Component root + table name | The name is echoed in every event so coordinators can route by table |
| `data-ln-data-table-selectable` | wrapper | Enables checkbox selection | Selection adds bytes (events, state, select-all logic); opt-in keeps simple tables simple |
| `data-ln-data-table-body` | `<tbody>` | Explicit render target | Falls back to first `<tbody>`; explicit attribute lets you have multiple tbodies and choose one |
| `data-ln-data-table-search` | `<input>` | Search input | Puts the search field anywhere in the DOM — toolbar, sidebar, modal — without changing the JS |
| `data-ln-data-table-clear-all` | `<button>` | Clears all column filters | Typically lives in the empty-filtered template; can also live in a sticky toolbar |

### Column headers — on `<th>` and its descendants

| Attribute | On | Description | Why it exists |
|---|---|---|---|
| `data-ln-col="field"` | `<th>` | Column → data field binding | Without this, the column has no field to sort/filter; useful for presentational columns like "Actions" |
| `data-ln-col-sort` | `<button>` | Sort toggle button | The button (not the `<th>`) is the click target so the rest of the header stays passive — clicks on the label, on a tooltip, on padding don't trigger sort |
| `data-ln-col-filter` | `<button>` | Filter dropdown trigger | Same reasoning — separate button so the header can carry both sort and filter without ambiguity |
| `data-ln-col-select` | `<th>` | Select-all column | Auto-creates a checkbox if the `<th>` is empty; lets you provide your own checkbox if you want a custom layout |
| `data-ln-sort-icon="none\|asc\|desc"` | `<svg>` inside sort button | Identifies which icon to show in which state | Three icons in DOM, CSS picks one based on `<th>` class — no JS swap, no flicker |

### Row template — inside `<template data-ln-template="{table}-row">`

| Attribute | On | Description | Why it exists |
|---|---|---|---|
| `data-ln-row` | `<tr>` | Row click target + selection target | Marks "this is a data row" — distinguishes from spacer rows or empty-state rows |
| `data-ln-row-id` | `<tr>` | Set by JS from `record.id` | Used internally for selection tracking and exposed in events |
| `data-ln-row-select` | `<input type="checkbox">` | Row selection checkbox | Selection only activates if `selectable` is set on the root |
| `data-ln-row-action="name"` | `<button>` | Row action button | Decouples "what action" (markup) from "what it does" (coordinator) |
| `data-ln-cell="field"` | any element inside the row | JS fills `textContent` | Most cells are simple text — this is the 90% case |
| `data-ln-cell-attr="field:attr"` | any element | JS sets attributes | For setting `href`, `src`, `data-*` from record values; comma-separated for multiple |

`data-ln-cell` and `data-ln-cell-attr` use `setAttribute` /
`textContent` — never `innerHTML`. If a record contains HTML, it is
escaped, not interpreted. If you need rich content per cell, generate
it as plain text in the coordinator before dispatching `set-data`, or
dispatch a `row-click` and open a modal — never inject HTML strings.

### Footer — anywhere inside the component root

| Attribute | On | Description | Why it exists |
|---|---|---|---|
| `data-ln-data-table-total` | `<span>` | Total record count | Always shown; populated even on empty data (renders `0`) |
| `data-ln-data-table-filtered` | `<span>` | Filtered count | Only meaningful when filters reduce the count |
| `data-ln-data-table-filtered-wrap` | parent of `-filtered` | Wrapper | Lets you hide the entire " · 12 filtered" fragment, including punctuation |
| `data-ln-data-table-selected` | `<span>` | Selected row count | Only relevant on selectable tables |
| `data-ln-data-table-selected-wrap` | parent of `-selected` | Wrapper | Same wrap pattern — hides leading punctuation when nothing is selected |

### Filter dropdown — inside the `column-filter` template

| Attribute | On | Description |
|---|---|---|
| `data-ln-filter-search` | `<input>` inside dropdown | Filters the option list within the dropdown (hidden if ≤8 values) |
| `data-ln-filter-options` | `<ul>` inside dropdown | Container the component populates with `<li><label><input type="checkbox">…</label></li>` |
| `data-ln-filter-clear` | `<button>` | Clears just this column's filter |

## Events

All events bubble from the `[data-ln-data-table]` element. Listen at
`document` level for cross-component coordination, or directly on the
table for tight scoping.

### Emitted by the table

| Event | Dispatched when | `detail` | Cancelable | Common consumer |
|---|---|---|---|---|
| `ln-data-table:request-data` | Init, after sort/filter/search/clear | `{ table, sort, filters, search }` | no | Coordinator — fetches data and dispatches `set-data` back |
| `ln-data-table:rendered` | After rows rendered into `<tbody>` | `{ table, total, visible }` | no | Layout (e.g., scroll-to-top, log) |
| `ln-data-table:sort` | User clicks sort button | `{ table, field, direction }` (`direction: "asc" \| "desc" \| null`) | no | Analytics, persistence |
| `ln-data-table:filter` | User changes a column's filter | `{ table, field, values }` | no | Analytics, URL sync |
| `ln-data-table:clear-filters` | User clicks a `[data-ln-data-table-clear-all]` button | `{ table }` | no | URL sync |
| `ln-data-table:search` | Every keystroke in `[data-ln-data-table-search]` | `{ table, query }` | no | Analytics |
| `ln-data-table:row-click` | User clicks a row (not on a button/link/checkbox) | `{ table, id, record }` | no | Open detail modal, navigate |
| `ln-data-table:row-action` | User clicks a `[data-ln-row-action]` button | `{ table, id, action, record }` | no | Open edit modal, dispatch delete confirm |
| `ln-data-table:select` | Selection changes (single row or select-all) | `{ table, selectedIds, count }` | no | Bulk-action toolbar |
| `ln-data-table:select-all` | Select-all checkbox toggled | `{ table, selected }` | no | Logging |

### Received by the table

| Event | Dispatched by | `detail` | Effect |
|---|---|---|---|
| `ln-data-table:set-data` | Coordinator | `{ data: [], total, filtered, filterOptions? }` | Renders rows; updates footer; refreshes filter options |
| `ln-data-table:set-loading` | Coordinator | `{ loading: Boolean }` | Toggles `ln-data-table--loading` class |

`set-data.detail.data` is the array of records to render *right now* —
already sorted, filtered and searched by the coordinator. `total` is the
unfiltered total (so the footer can say "12 of 1,000"). `filtered` is
the count after filters but before any pagination. `filterOptions`, if
provided, is an authoritative `{ field: [values] }` object replacing
the cache for those fields — see the architecture doc for details on
the additive vs authoritative flow.

## API (component instance)

Every initialized table exposes `el.lnDataTable` on the root element.
Reads are encouraged for inspection; mutations should go through events.

| Property | Type | Description |
|---|---|---|
| `isLoaded` | `boolean` | `true` once the first `set-data` has been received |
| `totalCount` | `number` | Last-known total (across all data) |
| `visibleCount` | `number` | Last-known visible (after filter) count |
| `currentSort` | `{ field, direction } \| null` | Current sort intent, mirrored from the last `sort` event |
| `currentFilters` | `{ [field]: string[] }` | Active filter values per column |
| `currentSearch` | `string` | Current search input value |
| `selectedIds` | `Set<string>` | IDs of selected rows (string keys, even if record IDs are numeric) |
| `selectedCount` | `number` | Computed from `selectedIds.size` |
| `destroy()` | method | Tears down listeners, closes filter dropdown, clears cached data |

`window.lnDataTable(rootEl)` initializes a freshly inserted DOM subtree.
The MutationObserver picks up most cases automatically — call it
explicitly when you've inserted markup via `innerHTML` and want immediate
init.

## Examples

### Minimal — read-only listing, no selection

A table that shows data and lets the user sort one column. No checkboxes,
no filters, no search, no actions. Use this when the table is purely
informational.

```html
<section data-ln-data-table="logs">
	<table>
		<thead>
			<tr>
				<th data-ln-col="time">
					Time
					<button data-ln-col-sort aria-label="Sort">
						<svg class="ln-icon" aria-hidden="true" data-ln-sort-icon="none"><use href="#ln-arrows-sort"></use></svg>
						<svg class="ln-icon" aria-hidden="true" data-ln-sort-icon="asc"><use href="#ln-arrow-up"></use></svg>
						<svg class="ln-icon" aria-hidden="true" data-ln-sort-icon="desc"><use href="#ln-arrow-down"></use></svg>
					</button>
				</th>
				<th data-ln-col="message">Message</th>
			</tr>
		</thead>
		<tbody data-ln-data-table-body></tbody>
	</table>

	<template data-ln-template="logs-row">
		<tr data-ln-row>
			<td data-ln-cell="time"></td>
			<td data-ln-cell="message"></td>
		</tr>
	</template>
</section>
```

### Server-driven — typical Laravel listing

The most common production shape. Coordinator listens for `request-data`,
calls a Laravel endpoint, dispatches `set-data` back. Search and filters
are part of the request payload — the server does the work, the table
just renders.

```javascript
const tableEl = document.querySelector('[data-ln-data-table="documents"]');

document.addEventListener('ln-data-table:request-data', async function (e) {
	if (e.detail.table !== 'documents') return;

	tableEl.dispatchEvent(new CustomEvent('ln-data-table:set-loading', {
		detail: { loading: true }
	}));

	const res = await fetch('/api/documents?' + new URLSearchParams({
		sort: e.detail.sort ? e.detail.sort.field : '',
		direction: e.detail.sort ? e.detail.sort.direction : '',
		filters: JSON.stringify(e.detail.filters || {}),
		q: e.detail.search || ''
	}));
	const result = await res.json();

	tableEl.dispatchEvent(new CustomEvent('ln-data-table:set-data', {
		detail: {
			data: result.data,
			total: result.meta.total,
			filtered: result.meta.filtered,
			filterOptions: result.meta.filterOptions
		}
	}));

	tableEl.dispatchEvent(new CustomEvent('ln-data-table:set-loading', {
		detail: { loading: false }
	}));
});
```

### Client-side cache via ln-store

When data is small enough to live in IndexedDB. The coordinator reads
from the store, applies sort/filter/search synchronously, and feeds
results back. Search is instant because there's no network round-trip.

```javascript
document.addEventListener('ln-data-table:request-data', async function (e) {
	if (e.detail.table !== 'documents') return;

	const store = document.querySelector('[data-ln-store="documents"]');
	const records = await store.lnStore.query({
		sort: e.detail.sort,
		filters: e.detail.filters,
		search: e.detail.search
	});

	tableEl.dispatchEvent(new CustomEvent('ln-data-table:set-data', {
		detail: { data: records, total: records.totalCount, filtered: records.length }
	}));
});
```

### Static — data already in the page

When the page already ships with the full data set (e.g., a small
admin dashboard rendered server-side). The coordinator filters and
sorts an in-memory array.

```javascript
const ALL = window.__INITIAL_DATA__ || [];

document.addEventListener('ln-data-table:request-data', function (e) {
	if (e.detail.table !== 'docs') return;
	let result = ALL.slice();

	if (e.detail.search) {
		const q = e.detail.search.toLowerCase();
		result = result.filter(r => r.title.toLowerCase().includes(q));
	}

	if (e.detail.sort) {
		const dir = e.detail.sort.direction === 'desc' ? -1 : 1;
		const f = e.detail.sort.field;
		result.sort((a, b) => (a[f] > b[f] ? 1 : -1) * dir);
	}

	tableEl.dispatchEvent(new CustomEvent('ln-data-table:set-data', {
		detail: { data: result, total: ALL.length, filtered: result.length }
	}));
});
```

### Bulk-action toolbar reacting to selection

Selection events drive a bulk toolbar that's only visible when at least
one row is selected. The component does not own the toolbar — it just
announces selection changes.

```javascript
const toolbar = document.getElementById('bulk-toolbar');

document.addEventListener('ln-data-table:select', function (e) {
	if (e.detail.table !== 'documents') return;
	toolbar.classList.toggle('hidden', e.detail.count === 0);
	toolbar.querySelector('[data-count]').textContent = e.detail.count;
});

document.getElementById('bulk-delete').addEventListener('click', function () {
	const ids = Array.from(tableEl.lnDataTable.selectedIds);
	// fetch DELETE /api/documents { ids } ...
});
```

## Common mistakes

### Mistake 1 — Putting the row markup directly in `<tbody>`

```html
<!-- WRONG -->
<tbody data-ln-data-table-body>
	<tr data-ln-row>
		<td data-ln-cell="title"></td>
	</tr>
</tbody>
```

The component clears `<tbody>` on every `set-data` and replaces it with
clones from the `{table}-row` template. Static rows are wiped on the first
render. Put the row layout inside `<template data-ln-template="{name}-row">`.

### Mistake 2 — Mismatched `data-ln-col` and `data-ln-cell` field names

```html
<!-- WRONG — column says "title", cell says "name" -->
<th data-ln-col="title">Title</th>
...
<td data-ln-cell="name"></td>
```

The cell stays empty because the record has no field called `name` (or
because the sort request sends `field: "title"` but the data was
prepared under `name`). The two strings must match exactly — they're
the same identifier. Pick a convention (the API field name) and keep
both sides aligned.

### Mistake 3 — Using a single sort icon and swapping the `<use href>` in JS

```html
<!-- WRONG -->
<button data-ln-col-sort>
	<svg class="ln-icon"><use href="#ln-arrows-sort"></use></svg>
</button>
```

Tempting because "one icon, JS swaps the href on sort." It works visually
but it's the wrong layer doing the swap — JS is mutating SVG attributes
to convey state that already lives in a class on the `<th>`. The library
shape is three sibling icons with `data-ln-sort-icon` values; CSS shows
the right one. Use the three-icon pattern.

### Mistake 4 — Building rows in JS instead of using the template

```javascript
// WRONG
data.forEach(r => {
	tbody.innerHTML += `<tr><td>${r.title}</td></tr>`;
});
```

This bypasses the template, opens an XSS hole on `r.title`, and
duplicates row layout in the script file (now you have to redeploy JS
to add a column). Use `set-data`, let the component render via the
template — `data-ln-cell` uses `textContent` and is XSS-safe.

### Mistake 5 — Calling internal methods from outside

```javascript
// WRONG
tableEl.lnDataTable._renderRows();
tableEl.lnDataTable.currentFilters['status'] = ['Approved'];
```

Methods prefixed with `_` are internal. Mutating `currentFilters` does
not trigger a re-request. The contract for state changes is events:
dispatch `set-data` to change rendered data, dispatch `set-loading` to
toggle the loading state. Reads (`isLoaded`, `selectedIds`) are fine.

## Related

- **`ln-store`** — IndexedDB-backed local cache; common upstream when
  the table coordinator pulls from a store rather than a server.
- **`ln-table`** — sister component that parses an existing populated
  `<tbody>` instead of being fed via events. Use it for static data
  rendered server-side that just needs sort/responsive behaviors.
- **`ln-toast`** — typical downstream of `ln-data-table:row-action`
  for "Deleted" / "Updated" notifications.
- **`ln-modal`** — typical downstream of `ln-data-table:row-click`
  or `row-action` for detail / edit dialogs.
