    ashlar-gui v1.2.0

    Overview

    A 152-line wrapper around TomSelect that exists for one reason: the platform's native <select> is usable but not enough. It cannot search 200 options, cannot tag, cannot create on the fly, cannot render multi-select chips that survive a finger on a touchscreen. TomSelect solves all of that. ln-select adds the smallest amount of glue — auto-init via MutationObserver, JSON-config from a single attribute, form-reset hookup, graceful degradation when TomSelect is missing — so the form layer (ln-form, ln-validate, ln-autosave) can compose with TomSelect through the platform's own change event without ever knowing TomSelect exists.

    The contract is one sentence: put data-ln-select on a <select> and TomSelect upgrades it; remove the element from the DOM and the instance is destroyed. Configuration travels as JSON inside the attribute value (data-ln-select='{"create": true}'), merged on top of seven sensible defaults. The TomSelect instance itself lives in a module-scoped WeakMap, accessible via window.lnSelect.getInstance(element) — never on el.lnSelect directly.

    For the full guide see the ln-select README and the architecture reference.

    Anatomy

    A select-enhanced field is one element — a <select> with data-ln-select. Optional JSON in the attribute value overrides any of the wrapper's seven defaults.

    <select data-ln-select name="role">
    <option value="">Select a role…</option>
    <option value="admin">Administrator</option>
    <option value="editor">Editor</option>
</select>

<select data-ln-select='{"create":true}' name="tags[]" multiple>
    <option value="design">design</option>
    <option value="engineering">engineering</option>
</select>

    The host — <select data-ln-select>

    The init scan filters by select[data-ln-select]. Putting the attribute on a <div>, <input>, or any other tag does nothing — silent skip, no console.warn. The element renders as written.

    The configuration — JSON in the attribute value

    Empty value or boolean attribute uses defaults. Non-empty value is parsed as JSON and merged on top:

    // Defaults the wrapper applies
{
    allowEmptyOption: true,    // honor <option value=""> as a real choice
    controlInput:     null,    // hide type-to-search input on single-select
    create:           false,   // disallow user-typed new options
    highlight:        true,    // highlight matched substring in dropdown
    closeAfterSelect: true,    // close dropdown after each pick
    placeholder:      <element placeholder> || 'Select...',
    loadThrottle:     300,     // ms throttle for async load callback
}

    Invalid JSON logs [ln-select] Invalid JSON in data-ln-select attribute: and the wrapper falls back to {} — degraded, not crashed.

    What state goes where

    "Should this <select> be enhanced?" lives in data-ln-select presence. "Currently selected value" lives in <select>.value (single) or <select>.selectedOptions (multiple) — TomSelect writes through. The TomSelect instance itself lives in a module-scoped WeakMap, accessed via window.lnSelect.getInstance(el). The visible chip pills / dropdown panel are TomSelect's own DOM (.ts-wrapper sibling of the <select>).

    Why WeakMap instead of el.lnSelect?

    Most ln-ashlar components store their instance directly on the host element — el.lnToggle, el.lnTabs, el.lnAutosave. ln-select deliberately does not.

    TomSelect mutates the host <select> heavily: hides it with inline display: none, inserts a sibling .ts-wrapper DOM, intercepts every option / change / click. Storing a wrapper instance on select.lnSelect would mean every TomSelect-internal query (select.options, jQuery-style $select, Object.keys(select)) would have to skip past an ln-ashlar-specific property. The WeakMap keeps the instance findable for the wrapper's API while leaving the element's own attribute / property surface untouched.

    The WeakMap also enables GC of detached elements automatically. When a <select> is removed from the DOM and there are no other references to it, the WeakMap entry becomes eligible for garbage collection along with the element.

    The trade-off is that el.lnSelect is undefined. Consumers must go through window.lnSelect.getInstance(el) — which is the load-bearing detail that makes ln-autosave's dead-code branch (if (restored[k].lnSelect && restored[k].lnSelect.setValue) at ln-autosave.js:97-99) never execute. The synthetic-change dispatch one line above does the actual work.

    Live demo — single select with search

    The dropdown is searchable: type "ad" to filter to "Administrator". Keyboard (arrow keys, enter), mouse, or touch — all three commit the selection by writing to the underlying <select>.value and firing change.

    Open DevTools and inspect the <select> — select.value mirrors whatever you picked. The visible chip is TomSelect's .ts-wrapper sibling.

    Live demo — multi-select with cap (maxItems: 3)

    The multiple attribute on the <select> turns the picker into chip pills. The {"maxItems":3} JSON config caps the number of selections at 3 — after the third, the dropdown stops accepting new picks until the user removes one.

    The [] in the name attribute is the Laravel / Rails / Express convention for "this is an array on the server side." When the form serializes, the field appears as repeated tags[]=design&tags[]=engineering.

    Live demo — tagging (create: true)

    With {"create":true}, the user can type a value that is not in the option list and press Enter to add it. TomSelect appends a new <option> to the underlying <select> as the user creates, so the new tag is in the serialized payload alongside the existing ones.

    Try typing rust and pressing Enter. Then inspect the <select> in DevTools — the new <option value="rust"> is there.

    Live demo — composes with ln-validate

    The required HTML attribute on the <select> is honored by dom.checkValidity() — the value the user picks is what gets validated. ln-validate wires a change listener; TomSelect commits selections via change on the underlying <select>; validation runs on every selection commit. Zero coordinator wiring.

    • Please choose a country

    Pick a country and then re-select the empty option. The error appears immediately. Pick a real country again, the error clears. The change listener in ln-validate sees the platform event regardless of which UI committed it.

    Live demo — composes with ln-form (fill, reset, submit)

    ln-form.fill() writes select.value = "..." via populateForm() and dispatches synthetic change on every restored field. TomSelect's internal change listener catches the synthetic event and re-syncs its visible UI. The integration works because TomSelect listens for the platform's change event, not for a custom ln-form:fill event.

    • Pick a language

    Click Fill — both selects animate to the filled values. Click Reset — the wrapper's reset hookup runs tomSelect.clear() + clearOptions() + sync() on the next tick (after the platform finishes resetting the underlying <select>).

    Live demo — composes with ln-autosave (restore via synthetic change)

    Pick values, then reload the page. ln-autosave reads the draft from localStorage, calls populateForm() which writes select.value = ... and dispatches synthetic change. TomSelect's listener fires and re-renders the chip pills to match. The dead-code branch at ln-autosave.js:97-99 (restored[k].lnSelect.setValue) does NOT execute — instances are in the WeakMap, not on the element. The synthetic-change path (one line above, at ln-autosave.js:94-95) is what actually drives TomSelect's UI sync.

    Inspect localStorage after picking — the key is ln-autosave:<path>:select-demo with the serialized values.

    Reading and writing programmatically

    For most reads, the platform value is the right choice — TomSelect keeps the underlying <select> in sync, so select.value is always current.

    const select = document.getElementById('role');

// Native — works whether TomSelect is upgraded or not
console.log(select.value);              // "admin"

// Via TomSelect (only after upgrade)
const ts = window.lnSelect.getInstance(select);
if (ts) {
    console.log(ts.getValue());         // "admin"
    console.log(ts.items);              // ["admin"]
}

    For writes, use ts.setValue() or dispatch change manually after a direct .value write — otherwise TomSelect's UI does not re-render.

    // RIGHT — via TomSelect's API (preferred)
const ts = window.lnSelect.getInstance(select);
if (ts) ts.setValue('editor');

// RIGHT — direct write + synthetic change (works because TomSelect listens for change)
select.value = 'editor';
select.dispatchEvent(new Event('change', { bubbles: true }));

// WRONG — direct write without dispatching change (TomSelect UI stale)
select.value = 'editor';

    API — window.lnSelect

    MethodDescription
    initialize(element)Upgrade the element. Idempotent — repeat calls are no-ops. Used internally by the observer; rarely needed in consumer code.
    destroy(element)Tear down the TomSelect instance, unwire the form-reset listener, delete the WeakMap entry. Used internally by the observer's removedNodes path.
    getInstance(element)Return the TomSelect instance, or undefined if the element is not upgraded. The single source of truth for finding the wrapper.

    There is no reinit(), no getAll(), no event emitter. The API is intentionally narrow. The js/ln-ajax/README.md:95 example that calls window.lnSelect.reinit() is a documentation bug — the MutationObserver covers AJAX inserts automatically, no manual reinit needed.

    Configuration — defaults vs user

    The merge is shallow object spread. User-provided keys overwrite the wrapper's defaults; nested objects (render, plugins) are replaced wholesale, not merged.

    DefaultWhy this default
    allowEmptyOption: trueHonor <option value="">Select…</option> as a real selectable choice.
    controlInput: nullHide the type-to-search input on single-select. Set to '<input>' to re-enable.
    create: falseDefault is "fixed list of options." Override to true for tag inputs.
    highlight: trueBold the matched substring in the dropdown.
    closeAfterSelect: trueSingle-pick: close after pick. Multi-pick: also close after each pick. Override to false for power-user multi-select flows.
    placeholderReads the placeholder attribute on the <select> if set; otherwise 'Select...'.
    loadThrottle: 300Async load callback throttle in ms.

    Every other TomSelect option is reachable via the JSON config. Function-valued options (load, render, onChange) cannot live in JSON — for those, use the wrapper's defaults via the attribute and reach for the TomSelect instance via getInstance(el) for any imperative customization.

    Common mistakes

    Putting data-ln-select on a non-<select>

    <!-- Silently ignored. The init scan filters by select[data-ln-select]. -->
<div data-ln-select>Choose…</div>
<input data-ln-select>

    Forgetting to load TomSelect before ln-ashlar

    The wrapper checks window.TomSelect at script load (lines 15-20 of ln-select.js). If TomSelect hasn't been parsed yet, the wrapper installs a no-op API and exits. Loading TomSelect afterwards does NOT retroactively upgrade — the observer was never registered.

    <!-- RIGHT — TomSelect first, ln-ashlar second -->
<link rel="stylesheet" href="tom-select/dist/css/tom-select.css">
<script src="tom-select/dist/js/tom-select.complete.min.js"></script>
<link rel="stylesheet" href="dist/ln-ashlar.css">
<script src="dist/ln-ashlar.iife.js"></script>

    Direct select.value = ... without dispatching change

    TomSelect's visible UI does not update; ln-validate does not validate; ln-autosave does not save. Either use ts.setValue() or dispatch change manually.

    Disabling the underlying <select> directly

    // WRONG — TomSelect's chip pill / dropdown stays interactive
select.disabled = true;

// RIGHT — use TomSelect's API
const ts = window.lnSelect.getInstance(select);
if (ts) ts.disable();   // visual disable + sets <select>.disabled

    Calling window.lnSelect.reinit()

    There is no reinit() on the API surface — only initialize, destroy, and getInstance. The MutationObserver covers AJAX inserts automatically. The example in js/ln-ajax/README.md:95 is a documentation bug; ignore it.

    Form-reset clears user-created options too

    The wrapper's reset handler runs clear + clearOptions + sync in the next tick. Options created via {"create":true} are NOT preserved — reset means "back to the form's initial state." If you need to preserve user-created options, use <button type="button"> + manual ts.clear() instead of type="reset".

    Graceful degradation

    If window.TomSelect does not exist, the wrapper logs [ln-select] TomSelect not found. Load TomSelect before ln-ashlar. and installs a no-op window.lnSelect = { initialize, destroy, getInstance }. Every <select data-ln-select> continues to work as a plain native dropdown. The page does not crash; the attribute becomes a noop marker.

    This is the right default for cases where the TomSelect bundle was forgotten, blocked by a CSP, or excluded from a minimal build. The page degrades visually (no search, no chip pills, no tagging) but the underlying form stays functional.

    Lifecycle

    TriggerWhat the wrapper does
    Script load with window.TomSelect presentCreate WeakMap, schedule initializeAll + observeDOM at DOMContentLoaded (or immediately if already loaded).
    Script load with window.TomSelect missingLog console.warn, install no-op API, exit.
    data-ln-select added to existing <select>Observer fires on attribute mutation, calls initializeSelect (idempotent).
    New <select data-ln-select> added to DOMObserver fires on addedNodes, walks subtree, calls initializeSelect for each match.
    <select data-ln-select> removed from DOMObserver fires on removedNodes, walks subtree, calls destroySelect for each match.
    Form fires resetWrapper's listener runs; setTimeout(0) defers clear + clearOptions + sync until after the platform's reset completes.
    User picks an optionTomSelect commits to <select>.value, fires change. ln-validate, ln-autosave, project listeners react via the platform event.
    data-ln-select attribute removedObserver fires, but the matcher select[data-ln-select] fails. The instance stays alive — clean teardown requires window.lnSelect.destroy(el) before attribute removal.

    Peer dependency

    TomSelect is a peer dependency, not a bundled one. Load both CSS and JS before ln-ashlar:

    <!-- TomSelect CSS -->
<link rel="stylesheet" href="tom-select/dist/css/tom-select.css">

<!-- TomSelect JS — must come BEFORE ln-ashlar -->
<script src="tom-select/dist/js/tom-select.complete.min.js"></script>

<!-- ln-ashlar -->
<link rel="stylesheet" href="dist/ln-ashlar.css">
<script src="dist/ln-ashlar.iife.js"></script>

    The wrapper does not re-document TomSelect — for option reference, render hooks, plugin list, async load callback signature, see the TomSelect docs.
