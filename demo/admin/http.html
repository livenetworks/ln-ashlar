    HTTP

    Two-path HTTP coordinator. Path A wraps window.fetch for transparent URL-keyed dedup of GET / HEAD. Path B is a ln-http:request event with a string key for explicit-cancel of any method (including POST). The two paths share state-tracking maps but otherwise do not interact.

    What it does (short version)

    • Path A — wraps window.fetch. Idempotent dedup on GET / HEAD: a new request to the same URL aborts the previous one. POST / PUT / PATCH / DELETE are NEVER auto-cancelled — losing one of two POSTs would silently drop user-intent data.
    • Path B — listens for ln-http:request on document. If the dispatch carries detail.key, a previous in-flight request with the same key is aborted regardless of method. Response fires as ln-http:response on the original target with the raw Response; the consumer calls .json().
    • Public API — lnHttp.cancel(url), lnHttp.cancelByKey(key), lnHttp.cancelAll(), lnHttp.inflight, lnHttp.destroy().
    • What ln-http does NOT do: no header injection, no auth/CSRF handling, no body shaping, no JSON parsing, no retry, no timeout, no progress events. The consumer composes the request; ln-http coordinates cancellation.

    The demos below run against a page-local mock fetch — no server is required. The mock simulates network delay so dedup behavior is observable.

    Demo 1 — Search-as-you-type (Path A)

    Type rapidly. Each keystroke fires a GET to /api/search?q=…. Different URLs (different q) run independently; same URLs (e.g. backspacing to a previously-typed query) collapse — the older aborts. The log shows abort errors with AbortError; only the latest typed query's response renders.

      Demo 2 — Drag-reorder POST with explicit key (Path B)
      
	
      
		
      Click Move up / Move down rapidly. Each click dispatches a POST ln-http:request with key: 'reorder-list-1'. Path A would NOT cancel POSTs by URL; Path B aborts the previous in-flight one because the keys match. The log shows that fast clicks produce only one ln-http:response at the end.

      ln-http vs ln-ajax
      
	
      
		
      
		
      
			
      ln-httpln-ajax
      
			
				
      PurposeTransport coordination (dedup, cancel)HTML page navigation
      
				
      Wraps window.fetch?Yes (Path A) — transparentNo — calls native fetch
      
				
      Event APIln-http:request → ln-http:response / ln-http:error (Path B)ln-ajax:before-start / :start / :complete / :error
      
				
      Per-element attributeNone (global service)data-ln-ajax on element / link / form
      
				
      Auto-cancelGET/HEAD on same URL (Path A); any method by key (Path B)None
      
				
      Header injectionNoneAdds X-Requested-With + X-CSRF-TOKEN from meta
      
				
      Body handlingForwarded raw — consumer composesBuilds FormData from form, appends _token
      
				
      Response parsingNone — forwards raw Response.json() on the response
      
				
      Visual feedbackNoneSpinner + ln-ajax--loading class on element
      
				
      Use whenYou want dedup / cancel coordination across raw fetch callsYou want links / forms to swap server-rendered HTML without a full page load
      
			      
		
      
		
      
		
      The two coexist. ln-ajax's internal fetch calls go through ln-http's wrapper transparently — its GET path inherits Path A dedup. Their concerns don't overlap.
