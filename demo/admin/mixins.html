Mixins Reference

Complete list of @include mixins from scss/config/_mixins.scss. Import with @use 'config/mixins' as *; and use in semantic selectors.

Override Architecture

ln-acme provides predefined mixins. Projects use those mixins in their SCSS selectors. When a project needs different behavior, it creates a project-level mixin with the same name that overrides the ln-acme version. The project SCSS selectors stay unchanged.

The golden rule: ZERO hardcoded colors

Mixins NEVER contain concrete color values (#fff, rgb(), hsl(0 0% 100%)). Every color reads from a CSS custom property token: var(--color-*). This is what makes the entire system work — colors are inherited through the CSS cascade, not baked into mixin definitions.

WRONG

// Hardcoded — breaks inheritance
.ln-modal header {
  background-color: hsl(var(--color-primary));
  border-bottom: none;
  color: #fff;
}

CORRECT

// Mixin reads tokens, project just uses it
.ln-modal header {
  @include panel-header;
}

// Want a different color? Override the token
.ln-modal {
  --color-primary: 280 60% 50%;
  // header automatically inherits
}

Three layers

LayerRoleExample
ln-acme (library) Defines the default mixin — reads from var(--token), never hardcodes @mixin panel-header { background-color: hsl(var(--color-primary)); color: hsl(var(--color-white)); }
Project (consumer) Uses the mixin on semantic selectors. For different colors, overrides the CSS variable on a parent, not the mixin .ln-modal header { @include panel-header; }
.ln-modal { --color-primary: 280 60% 50%; }
Project override (rare) Redefines the mixin only when structure needs to change (layout, padding, border-radius) — still reads from tokens Project's own @mixin panel-header { ... }

When to use which approach

NeedApproachExample
Different color Override CSS variable on parent element .ln-modal { --color-primary: 280 60% 50%; }
Different structure (padding, layout, shape) Override the mixin in project _overrides.scss @mixin panel-header { @include px(2rem); ... }
Both Override mixin for structure + CSS variable for color Mixin override + .parent { --color-primary: ...; }

Example: overriding btn in a project

ln-acme default

// scss/config/_mixins.scss (ln-acme)
@mixin btn {
  @include flex;
  @include items-center;
  @include px(1rem);
  @include py(0.5rem);
  @include text-sm;
  @include rounded-md;
  @include cursor-pointer;
  @include transition;
  border: none;
  outline: none;
  white-space: nowrap;
}

Project override

// project/scss/_overrides.scss
@mixin btn {
  @include inline-flex;
  @include items-center;
  @include px(1.5rem);
  @include py(0.75rem);
  @include text-base;      // larger text
  @include font-medium;
  white-space: nowrap;
}

// Project selectors — UNCHANGED
#save-user   { @include btn; }
#delete-user { @include btn; }
.form-actions button { @include btn; }

How to set up the override

// project/scss/main.scss
// 1. Import ln-acme (loads default mixins)
@use 'ln-acme/scss/ln-acme';

// 2. Import project overrides AFTER ln-acme
//    This file redefines mixins with the same name
@use 'overrides';

// 3. Import project pages (use the mixins — they get the overridden version)
@use 'pages/users';
@use 'pages/dashboard';

Container Queries

Parent declares the container context, child queries it with native @container. Never combine container-type with overflow: hidden on the same element.

MixinOutput
@include containercontainer-type: inline-size
@include container(name)container-type: inline-size; container-name: name
 
// Parent — declares the context
#sidebar { @include container(sidebar); }

// Child — queries the container
#sidebar nav {
  @container sidebar (min-width: 580px) {
    display: grid;
    grid-template-columns: repeat(2, 1fr);
  }
}

Collapsible

Grid-based expand/collapse animation. Uses grid-template-rows: 0fr/1fr transition.

MixinRole
@include collapsibleParent — display: grid with 0fr → 1fr transition. Toggled by .open class
@include collapsible-contentChild — overflow: hidden to clip content
 
.my-panel          { @include collapsible; }
.my-panel > section { @include collapsible-content; }

Buttons

Every <button> gets hover/active/focus/disabled effects out of the box from ln-acme. All states read from --color-primary. No classes needed.

MixinDescription
@include btnButton structure: inline-flex, padding, font-size, font-weight. Colors come from global styles
@include btn-colorsOnly the color states (reads --color-primary). Already applied globally to all <button>
@include close-buttonTransparent close button with hover-red effect

Live demo

Changing color — override --color-primary

Override --color-primary on the element or any parent. The button inherits the new color automatically.

/* Project variant — overrides --color-primary on the button */
.btn--danger  { --color-primary: var(--color-error); }

/* Parent context — all buttons inside change color */
.admin-panel  { --color-primary: var(--color-secondary); }

/* Semantic selector */
#delete-user  { --color-primary: var(--color-error); }

Custom color in a project

/* Define a project-specific color token */
:root { --color-brand: 280 60% 50%; }

/* Override --color-primary on the context */
.brand-section { --color-primary: var(--color-brand); }
/* All buttons inside .brand-section now use brand color */
