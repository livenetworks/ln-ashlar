Overview & Philosophy

Asynchronous list filtering in complex web layouts often creates tight coupling between search inputs and target containers. Hardcoding filtering logic into inputs breaks reusability, while forcing developers to write custom debounce hooks and DOM manipulation routines on every new page duplicates effort and introduces race conditions.

In ln-ashlar, this is solved by ln-search, a zero-dependency **Debounced Search Primitive**. It functions strictly as an **intent announcer**. When a user types, it debounces the keystrokes and dispatches a single, normalized, and cancelable event (ln-search:change) directly on the target container. It does not own data filtering itself, allowing tables, lists, or custom server-search pipelines to intercept the search intent or let it fall back to a highly optimized built-in DOM-walking filter.

The Filtering Intent Doctrine

A search primitive should only supervise the input channel—handling keystroke debouncing, input clearing, and browser back-button value restorations. It must remain decoupled from data rendering. The cancelable `ln-search:change` event is the sole contract.

Orthogonal Integration Lifecycles

  1. The Passive Path (Zero-JS Fallback): When no custom event listener is attached, the primitive performs a lightweight direct child DOM walk on the target list, tagging non-matching items with data-ln-search-hide="true", which are hidden instantly via global library utility CSS.
  2. The Active Path (Custom & Table Integration): Active components (such as ln-table or custom server-side search scripts) intercept ln-search:change, call e.preventDefault() to deactivate the default DOM walk, and run their own internal in-memory or remote query pipelines instead.
  3. The Restoration Tick: Form recovery (back-button navigation or pre-filled inputs) is intercepted at construction. The component schedules an initial search dispatch via queueMicrotask, ensuring that all listener components are completely initialized and wired before the filter executes.

For the full guide see the ln-search README and the architecture reference.

The chrome (border, focus ring, icon spacing, clear-button visibility) comes from @mixin form-input-icon-group applied via <label>:has(input[type="search"]) in scss/components/_form.scss. class="search" adds the compact delta (recessed fill, capped width, 1rem icons).

Input resolution priority

When data-ln-search is on a wrapper, the input is found in this order: [name="search"], input[type="search"], input[type="text"]. If you nest multiple inputs inside the wrapper, the first match wins.

Default vs. deep targeting

The default behavior iterates target.children — direct children only. For nested structures where items are not direct children, opt into a CSS-selector pass via data-ln-search-items="selector"; the component then runs target.querySelectorAll(selector) on every search.

What state goes where

"Should this input drive search?" lives on data-ln-search="targetId" (owned by the author). "Currently typed term" lives on input.value (owned by the platform). "Item is hidden by search" lives on data-ln-search-hide="true" on each non-matching child of the target (owned by ln-search's default path only — consumers that preventDefault() own this themselves). Active debounce timer lives on the JS instance.

detail.term is already lowercased and trimmed — listeners receive a normalized string. detail.targetId echoes the value of data-ln-search for delegate disambiguation. The event is dispatched on the target, not the input — consumers naturally listen on the thing they care about (the table, the list), not on the input that drives it. Bubbling-up to document works for global listeners.

Calling preventDefault() from any listener disables the component's default DOM walk. Whatever the consumer does instead — table filter, server request, custom DOM swap — is now its responsibility. ln-table uses this exact pattern.

Composition — ln-table integration

When data-ln-search targets a [data-ln-table] element, ln-table's constructor wires a listener at js/ln-table/ln-table.js:62-78 that calls preventDefault() and applies the term in its in-memory filter pipeline. No coordinator, no extra configuration:

<input type="search" placeholder="Search..." data-ln-search="my-table">

<div id="my-table" data-ln-table>
    <table>...</table>
</div>

See the Table and Table Filter demos for live integrations.

Composition — independence from ln-filter

ln-search and ln-filter operate on the same target without coordinating. Each manages its own hide attribute; an item is visible iff both are absent:

[data-ln-search-hide],
[data-ln-filter-hide] {
    display: none !important;
}

The CSS rules ship in the library defaults. Neither component knows about the other.

Composition — what ln-search does not compose with

ln-table consumes ln-search:change. In both SSR and data-driven modes, ln-table listens for ln-search:change on its own element, so placing data-ln-search="<tableId>" on the search input (or its label wrapper) is the supported, recommended pattern. The term it receives is joined with sort and column filters into the ln-table:request-data payload (which ln-store consumes in data-driven demos).

ln-form does not coordinate with ln-search. A search input inside <form data-ln-form> works the same as outside one. ln-search listens to input only — never keydown or submit. Pressing Enter triggers the form's native submit; the 150ms debounce timer may still be running. If you need Enter-to-search semantics, wire the form's submit handler directly. See the ln-search README "Common mistakes" item 2.

API

const el = document.querySelector('[data-ln-search]');

el.lnSearch.targetId;          // string — captured at construction
el.lnSearch.input;             // HTMLInputElement — the resolved input
el.lnSearch.itemsSelector;     // string | null — value of data-ln-search-items
el.lnSearch.destroy();         // remove listeners, clear timer, deregister

window.lnSearch(root);         // re-run init scan over `root` — only needed
                               //   for Shadow DOM / iframe roots that the
                               //   global MutationObserver cannot see

There is no search(term) or clear() method. Trigger search programmatically by writing input.value and dispatching an input event:

input.value = 'argentina';
input.dispatchEvent(new Event('input', { bubbles: true }));
// → triggers the debounced search 150ms later

Common mistakes

1. Listening for input directly to bypass debounce

Bypasses the component's own debounce — every keystroke fires your handler. Listen for ln-search:change on the target instead; you get the debounce for free, and e.detail.term is already normalized.

2. Pressing Enter inside a form and expecting search-only behavior

The component listens to input only, not to keydown or submit. Pressing Enter in a form submits the form (native behavior). Either keep the input outside a form, or wire the form's submit handler explicitly.

3. Setting input.value programmatically and expecting search to fire

The platform does NOT fire input on property writes. Dispatch the event yourself: input.dispatchEvent(new Event('input', { bubbles: true })). Or use [data-ln-form]'s ln-form:fill, which dispatches input on every populated field.

4. Two data-ln-search inputs targeting the same target

Each input has its own debounce, its own dispatch, its own state. Typing in one does NOT clear or sync the other. The listener on the target sees whichever input fired most recently. Sync manually at the document level if you need it, or — better — collapse to a single search box.

5. Using the removed data-ln-table-search attribute

data-ln-table-search has been removed. ln-table does listen for ln-search:change in both SSR and data-driven modes — drive an ln-table search box with data-ln-search="<tableId>" on the input or its label wrapper. That is the sole supported hook.

6. Adding the clear button after construction

The constructor finds the clear button via querySelector once. Adding one later via setAttribute or appendChild gives an unwired button. Include the clear button in the initial markup, or destroy and re-init.

7. Mutating data-ln-search to switch targets

The targetId is captured at construction. Mutating the attribute does not re-resolve. To switch targets: el.lnSearch.destroy(), change the attribute, then window.lnSearch(parent).
