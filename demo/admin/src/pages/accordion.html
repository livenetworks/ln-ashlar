Overview & Philosophy

In traditional frontend libraries, an accordion is a heavy, monolithic component that owns everything: click events, height transitions, keyboard accessibility (ARIA), active-panel state, and storage persistence. If you want a multi-open variant or different persistence, you have to pass complex configuration flags.

In ln-ashlar, an accordion is fundamentally different: it is a "Coordinator". A coordinator is a lightweight, stateless ruleset that mediates between independent, highly-specialized primitives. The ln-accordion component is only 38 lines of JavaScript, and it possesses no state of its own. It simply sits back, listens to the DOM, and coordinates two other completely independent players: ln-toggle (which manages binary state and ARIA) and the CSS .collapsible mixin (which manages transitions).

The Coordinator Doctrine

A coordinator never invokes instance methods directly on other components, nor does it maintain a JavaScript registry of their states. It listens for events bubbling up from the DOM, and applies decisions by writing standard HTML attributes back to the target elements. The HTML attribute is the sole contract.

The Mindset: Three Orthogonal Layers

Every accordion you see on this page is a perfect synchronization of three isolated concerns, layered gracefully on top of each other:

  1. The State Primitive (ln-toggle): Each accordion panel is an independent ln-toggle instance. It owns the binary open / close state, coordinates trigger buttons, synchronizes aria-expanded and aria-controls, and handles per-panel localStorage persistence (data-ln-persist). It knows absolutely nothing about other panels or the fact that it is inside an accordion.
  2. The Animation Engine (CSS .collapsible): The height transition is handled entirely in Vanilla CSS via the @mixin collapsible. It transitions grid-template-rows from 0fr to 1fr. No JavaScript timers, no inline element style hacks, and no framerate stuttering.
  3. The Coordinator (ln-accordion): This is the glue. It lives on the wrapper element and enforces a single, exclusive rule: "When one panel opens, all other open panels in this group must close."

How the Coordination Flows

When a user clicks a trigger to open a panel:

  1. ln-toggle intercepts the click, opens the target panel by setting its attribute to data-ln-toggle="open", and dispatches a bubbled ln-toggle:open CustomEvent.
  2. The event bubbles up to the [data-ln-accordion] wrapper. The coordinator intercepts it, queries the immediate subtree for other elements carrying [data-ln-toggle], and sets data-ln-toggle="close" on them.
  3. Each closed panel's own ln-toggle MutationObserver reacts to its attribute change, fires its standard before-close hooks, handles persistence updates, and collapses its content using CSS.

Why This Mindset is Superior

  • Zero-configuration Multi-open: If you want multiple panels to be open at the same time, you do not need to configure an options object or change JavaScript properties. You simply remove the coordinator wrapper (data-ln-accordion), and the individual ln-toggle panels work perfectly on their own.
  • Unified Execution: Because the coordinator triggers a close by writing to the attribute, the closing panel runs through the exact same state machine as a direct user click. Cancelable events (before-close) are respected, chevrons rotate back smoothly, and localStorage persistence is automatically updated.
  • Nestable by Nature: Since ownership is determined purely by DOM ancestry (using .closest('[data-ln-accordion]')), accordions can be nested inside accordions infinitely. The inner coordinator handles its group, the outer coordinator handles its group, and they remain completely oblivious to each other.

For a deeper dive into the architectural rationale, see the Data Flow Architecture, the ln-accordion README, and the architecture deep-dive.

Anatomy

The minimal shape. Every working accordion is a variation on this skeleton.

<ul data-ln-accordion>
    <li>
        <header data-ln-toggle-for="p1">
            Section 1
            <svg class="ln-icon ln-chevron"><use href="#ln-arrow-down"></use></svg>
        </header>
        <section id="p1" data-ln-toggle="open" class="collapsible">
            <article class="collapsible-body"><p>Content 1.</p></article>
        </section>
    </li>

    <li>
        <header data-ln-toggle-for="p2">
            Section 2
            <svg class="ln-icon ln-chevron"><use href="#ln-arrow-down"></use></svg>
        </header>
        <section id="p2" data-ln-toggle class="collapsible">
            <article class="collapsible-body"><p>Content 2.</p></article>
        </section>
    </li>
</ul>

The wrapper — <ul data-ln-accordion>

The wrapper is a plain marker — the attribute takes no value. <ul> because an accordion is, semantically, an unordered list of collapsible items. The wrapper is the listener for the bubbled ln-toggle:open event and the dispatch element for ln-accordion:change.

Each item — <li> with a trigger header and a panel

The trigger declares which panel it controls via data-ln-toggle-for="p1"; the panel declares its own id="p1". The pairing is by ID, not by DOM proximity — the <li> just keeps them visually grouped. The chevron icon inside the header is plain decoration; ln-toggle's default CSS rotates any .ln-chevron 180° when the controlled panel is open, driven by aria-expanded on the trigger.

The panel — <section data-ln-toggle class="collapsible">

Three concerns layered on one element. id="p1" is the lookup target for the trigger. data-ln-toggle creates an ln-toggle instance — its value ("open" or anything else) is the open/closed state. class="collapsible" is the CSS animation: grid-template-rows: 0fr by default, 1fr when the element carries .open (added by ln-toggle from its attribute).

Element type is <section>. It cannot be <main> — the HTML spec allows only one <main> per page.

The body — <article class="collapsible-body">

The collapse animation requires the parent (.collapsible) to have zero padding so it can animate to zero height cleanly. Padding and margins live one level deeper, on the body child. Putting padding on .collapsible directly leaves a thin always-visible strip when collapsed.

What state goes where

"Is this panel open?" lives on the panel's data-ln-toggle attribute (owned by ln-toggle). "Is the chevron rotated?" lives on the trigger's aria-expanded (owned by ln-toggle). "Did opening this panel close the others?" is transient — handled by ln-accordion reactively, with no stored state. The accordion's instance has two fields: dom and _onToggleOpen. Nothing else.

Example — Nested accordions

Nested accordions are supported. Events

The accordion dispatches one event per panel-open and one on destroy. Both bubble; both are non-cancelable; detail.target is the panel for :change and the wrapper for :destroyed.

EventBubblesCancelableDetailDispatched when
ln-accordion:changeyesno{ target: HTMLElement }After every ln-toggle:open is processed (siblings closed; :change dispatched even when nothing needed closing)
ln-accordion:destroyedyesno{ target: HTMLElement }Inside destroy()
 
document.addEventListener('ln-accordion:change', function (e) {
    console.log('Active panel:', e.detail.target.id);
});

The accordion does not listen to ln-toggle:close. Closing the only-open panel just leaves the accordion fully collapsed — that's intentional.

API

There is no open(panelId) or close(panelId) method. To programmatically change the active panel, set data-ln-toggle="open" directly on the target panel — ln-toggle's observer will apply the open state, the event will bubble up, and ln-accordion will close the previously-open siblings.

// Open panel2 — accordion sees the bubbled ln-toggle:open and closes panel1
document.getElementById('panel2').setAttribute('data-ln-toggle', 'open');

// Tear down the coordinator (children remain intact)
const acc = document.querySelector('[data-ln-accordion]');
acc.lnAccordion.destroy();   // dispatches ln-accordion:destroyed

Common mistakes

1. Padding on .collapsible directly

The collapse animation works by transitioning grid-template-rows from 0fr to 1fr. Padding on the parent leaves a visible strip when collapsed. Padding always goes on .collapsible-body.

<!-- WRONG -->
<section data-ln-toggle class="collapsible" style="padding: 1rem;">…</section>

<!-- RIGHT -->
<section data-ln-toggle class="collapsible">
    <article class="collapsible-body">…</article>
</section>

3. Putting data-ln-toggle on the trigger header

data-ln-toggle creates a panel; data-ln-toggle-for creates a trigger. They are mutually exclusive — one element is one or the other. Putting both on the header turns the header into a panel that controls itself.

<!-- WRONG -->
<li>
    <header data-ln-toggle data-ln-toggle-for="p1">Section 1</header>
    <section id="p1" class="collapsible">…</section>
</li>

4. Panels living outside the accordion subtree

The pairing between trigger and panel is by ID, so ln-toggle works fine across the wrapper boundary. But the accordion's listener is scoped to its own subtree — bubbled ln-toggle:open events from outside don't reach it, so the single-open rule won't fire. Keep panels inside the accordion's DOM.

<!-- WRONG — panels outside the accordion -->
<ul data-ln-accordion>
    <li><header data-ln-toggle-for="p1">Section 1</header></li>
</ul>
<section id="p1" data-ln-toggle class="collapsible">…</section>
