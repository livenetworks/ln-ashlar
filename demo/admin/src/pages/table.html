Table

Overview & Philosophy

In web application design, data grids can often be pre-rendered completely on the server. In these scenarios, introducing a heavy client-side JSON-to-HTML template rendering pipeline (such as ln-data-table) is unnecessary overhead. The DOM already contains the correct HTML rows on initial load.

In ln-ashlar, this is solved by ln-table, a highly-optimized **Server-Rendered Table Primitive**. Instead of generating rows dynamically, it parses existing DOM rows directly on paint. It layers ultra-fast, in-memory client-side sorting, column filtering, text searches, and virtual scroll on top of standard pre-rendered HTML, providing a fluid desktop-grid experience with zero database fetching overhead.

The Server-Rendered Integration

Unlike data tables (which consume raw JSON via event streams), ln-table expects standard rows (<tr>) already printed in <tbody> on page load. It acts as an upgrade node: parsing cell texts dynamically and hiding or sorting DOM elements in-place. The HTML DOM is the single source of truth.

Three Orthogonal Layers of HTML Table Enhancements

  1. The Pre-Rendered Markup (HTML DOM): Rows arrive completely rendered from the server. Cells can contain rich markup (links, pills, or custom badges) natively on initial paint.
  2. The Visual Class Hooks (CSS State): Visual indicators (such as sort states or popover filters) are managed by standard CSS classes. The button markup contains static icons; CSS manages what is shown.
  3. The Parsing Shell (ln-table): The script acts as an enhancement node. It scans cells, parses raw values (using optional data-ln-value attributes), manages in-memory sort orders, and hides rows dynamically using filter criteria.

How the Local Search and Sort Coherence Works

  1. On load, the script parses every row, mapping text strings or custom data-ln-value targets into highly-optimized search and sort indexes in memory.
  2. When a user types in a connected search bar (data-ln-search) or clicks a sort header (data-ln-sort), the component executes in-memory, local array filters instantly.
  3. The component updates table rows in-place (toggling visibility or re-ordering nodes in the DOM). If the visible row count exceeds 200, it automatically activates virtual scrolling.

var table = document.getElementById('demo-table');

table.addEventListener('ln-table:ready', function (e) {
    countEl.textContent = e.detail.total.toLocaleString() + ' products';
});

table.addEventListener('ln-table:filter', function (e) {
    countEl.textContent = e.detail.matched < e.detail.total
        ? e.detail.matched.toLocaleString() + ' / ' + e.detail.total.toLocaleString()
        : e.detail.total.toLocaleString() + ' products';
});

data-ln-value — sort raw value

When display text differs from the sort key:

<!-- Without: sorts by "1.250" (string) -->
<td>1.250</td>

<!-- With: sorts by 1250 (number) -->
<td data-ln-value="1250">1.250</td>

<!-- Date: sorts by Unix timestamp -->
<td data-ln-value="1700000000">15.11.2023</td>

Virtual Scroll

Activates when filtered rows exceed 200. Renders only visible rows + 15 buffer rows above/below. Deactivates automatically when rows drop below threshold. Column widths are locked on first parse to prevent layout shifts.

Dynamic Rows (AJAX)

If <tbody> is empty at init, a MutationObserver waits for rows to appear and parses them automatically.

// Set innerHTML directly — ln-table detects it
document.querySelector('#products tbody').innerHTML = generatedHtml;
