HTTP

Overview & Philosophy

In modern asynchronous web applications, managing network concurrency is highly complex. Rapid keystrokes inside search fields, quick clicks on list reordering controls, or multi-modal submissions often trigger overlapping HTTP requests. Without strict controls, these requests finish out-of-order, corrupting the UI with stale data ("race conditions") or executing duplicate transactions on the database.

In ln-ashlar, this is solved by ln-http, a lightweight, zero-dependency **HTTP Concurrency Coordinator**. Instead of requiring developers to pass manual AbortController signals inside every fetch, `ln-http` operates globally. It coordinates in-flight requests on two parallel channels (Path A and Path B) to handle transparent URL-keyed de-duplication and explicit key-based cancellations.

The Concurrency Doctrine

A concurrency coordinator does not inject credentials, authorize tokens, or format JSON bodies. It focuses strictly on request cancellations. It runs as a global utility capturing fetch promises and aborting obsolete ones, leaving custom headers and payloads fully to the caller. The browser's native AbortController is the underlying contract.

Two Parallel Concurrency Channels

  1. Path A (Transparent URL-Keyed GET de-duplication): Wraps the global window.fetch natively. Whenever a GET or HEAD request is fired, it scans in-flight records. If it finds a matching URL (e.g. during fast keystroke searches), it automatically aborts the previous request. POST, PUT, and DELETE methods are **never** auto-canceled here to prevent dropping user intent.
  2. Path B (Explicit Key-Based Event Cancellation): Listens for ln-http:request events bubbling from descendants. If the request carries an explicit string `key` (e.g. `reorder-list-1`), any matching in-flight request—regardless of HTTP method—is instantly aborted. This prevents double-click issues on destructive actions and lists reordering.

How the Cancellation Flow Coheres

When a de-duplicated request is canceled:

  1. A new request arrives on Path A or B, matching the active URL or key.
  2. The coordinator calls .abort() on the older request's `AbortController` instantly, and wipes its cache pointer.
  3. The browser terminates the older network socket immediately. The initial fetch promise rejects with an AbortError, which is safely captured and ignored by presenters, while the new request runs cleanly.

For the full guide see the ln-http README and the architecture reference.
