Overview & Philosophy

Understanding local-first architecture in isolation is difficult. True appreciation comes from seeing a fully-wired CRUD application working flawlessly in real-world scenarios: syncing data instantly across tabs, queuing actions while offline, and automatically resolving server version conflicts (409) without freezing the UI.

This demo page illustrates the ultimate **Local-First CRUD Pipeline** in ln-ashlar, orchestrating five independent, highly-specialized components through standard DOM event channels:

The Multi-Tab Coherence

Open this page in a second browser window side-by-side. Make a change in one window—it saves locally, pushes to the backend mock, and immediately broadcasts to the other window using delta events. The other tab’s data table repaints itself instantly. No manual polling or web socket servers needed.

Five Interlocking Primitive Players

  1. The Cache Store (ln-store): Renders nothing. Manages local IndexedDB storage, outbox write queues, delta sync loops, and optimistic states.
  2. The Presenter (ln-data-table): Displays the records. It triggers query fetches, processes local column filters/sorting, renders templates, and captures selection events.
  3. The Form Gate (ln-form): Manages inputs. It handles native validation, block submissions, bulk-fills from row selections, and serializes updates.
  4. The Focus Gate (ln-modal): Houses the editor form. It keeps focus trapped, responds to escape cancellations, and opens or closes on demand.
  5. The Safety Gate (ln-confirm): Prevents accidental row deletions. It gates trash clicks in-place, arming and releasing actions cleanly.

Telemetry and Tele-Coordinating

All operations are mediated by a single, lightweight coordinator script at the bottom of the page. The coordinator coordinates queries, intercepts store mutations to refresh tables, handles toast notifications, and resolves complex 409 server conflicts dynamically by feeding merged payloads back into form fields.
