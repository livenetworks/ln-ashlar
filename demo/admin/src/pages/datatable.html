Overview & Philosophy

Rendering heavy grids and lists in dynamic web applications is a common source of performance bottlenecks. Traditional approaches load all rows into the DOM instantly (causing paint lags on huge sets), or couple data-fetching logic directly to the table markup itself, which prevents testing and reuse.

In ln-ashlar, this is solved by ln-data-table, a highly-performant, event-driven **Presenter Component**. Instead of managing database calls or maintaining visual coupling, the table functions as a pure presentation and query shell. It outputs user sorting, searching, and filtering intents, and awaits datasets to render. It features built-in virtual scrolling (rendering only visible and buffer rows to support 10,000+ items seamlessly) and premium keyboard navigation.

The Presenter Doctrine

A presenter component never holds direct references to network services or database cached caches (like ln-store). It states what filter/sort actions occurred through bubbled request events, and receives rendered datasets asynchronously. The table is completely decoupled from the data layer.

Three Orthogonal Layers of Grid Performance

Slick grid presentation is achieved by strictly dividing structural markup, lifecycle queries, and visual updates:

  1. The Render Template (HTML <template>): Individual row layouts are written in standard HTML template tags, fully isolated from Javascript. Cells support safe {{ field }} text interpolation (XSS-safe by using native textContent) and dynamic attribute mapping.
  2. The Visual Class Hooks (CSS State): Visual indicators (such as sort states or filter overlays) are managed purely by toggling standard classes on header cells (e.g. .ln-sort-asc). The button markup contains three static icons; CSS determines which one to display.
  3. The Presentation Shell (ln-data-table): The script manages user interactions. It tracks scroll positions for virtual rendering, coordinates row selections, captures search bar input delays, and dispatches data intents.

How the Presentation Flow Coherence Works

  1. On load or user change, the table dispatches ln-data-table:request-data, carrying the current query parameters (sort field, filters, search string).
  2. An external coordinator (or local sync store) captures the intent, queries the cache (or backend API), and dispatches ln-data-table:set-data back.
  3. The component receives the payload, flushes old nodes, clones template elements, populates cells, and sets scroll bounds. Sighted tables load immediately.

For the full guide see the ln-data-table README and the architecture reference.

The component root — <section data-ln-data-table="...">

The attribute value is the table name, unique per page. It's echoed in every event the table dispatches (detail.table === 'documents') so a coordinator listening at document level can route by name. data-ln-data-table-selectable is opt-in — it switches on row checkboxes, select-all, and the ln-data-table:select event.

The toolbar — yours, the table just listens

The component does not build the toolbar. It only watches for one element: an <input> carrying data-ln-data-table-search. Each keystroke updates state, dispatches ln-data-table:search and triggers a fresh request-data. Place the input wherever it fits the page layout.

Column headers — <th data-ln-col="field">

The string after data-ln-col= is the data field name. It ties the column to the row template — <th data-ln-col="title"> matches {{ title }}. Without data-ln-col, the column has no field to sort or filter on; it's treated as presentational (the "Actions" column is the typical example).

The sort button — three icons, one source of truth

The sort button contains three sibling <svg> icons, distinguished by data-ln-sort-icon="none|asc|desc". All three sit in the DOM at all times. CSS shows exactly one based on a class on the ancestor <th>:

  • Default (no class) → the "none" icon (ln-arrows-sort) is visible.
  • <th class="ln-sort-asc"> → the "asc" icon (ln-arrow-up) is visible.
  • <th class="ln-sort-desc"> → the "desc" icon (ln-arrow-down) is visible.

The component never swaps an SVG, never replaces an icon URL, never sets inline styles. It toggles a class on the <th>; CSS does the rest. This is the cleanest example in the library of markup-driven state — every visual change is a class flip, and the icons live declaratively in HTML where you can read them.

The row template — <template data-ln-template="{name}-row">

The component clones this template once per record and fills cells: it natively parses double-curly braces like {{ title }} for dynamic text injection, and also supports data-ln-cell-attr="field:attr" for attributes (comma-separated for multiple). Why a <template> and not a static row? A static row would render empty in the initial paint. A <template> is inert until cloned — the component owns when rows appear. Why not build rows in JS? Because then the row layout lives in your script file and you'd have to redeploy JS to add a column.

Cell text is set via textContent, never innerHTML

This is intentional and load-bearing — textContent is XSS-safe. Records can contain any string, including <script> tags or attribute-injection payloads, and they will render as visible text rather than be parsed as markup. If you need rich content per cell, render plain text and use row-click to open a detail view.

The footer — count spans with hide-on-zero wrappers

Three optional spans report numbers: data-ln-data-table-total, data-ln-data-table-filtered, data-ln-data-table-selected. The "filtered" and "selected" spans live inside -wrap parents that get the hidden class when the count is zero — so the punctuation around them ("· 12 filtered") disappears with the value. No awkward " · 0 filtered" or empty leading separator.

Implementation Patterns & Copy-Paste Snippets

Use these production-ready code snippets to integrate ln-data-table in your next project. Simply copy/paste and adapt the identifiers to match your schema.

1. AJAX Server-Side Coordinator (Backend REST API)

This JS block registers the coordinator. It listens for sort, search, and filter intentions, makes an asynchronous API call, handles the visual loading indicators, and pushes the data back to the component.

(function () {
    const tableEl = document.getElementById('products-table');
    if (!tableEl) return;

    tableEl.addEventListener('ln-table:request-data', function (e) {
        if (e.detail.table !== 'products') return;

        // 1. Show loading state
        tableEl.dispatchEvent(new CustomEvent('ln-table:set-loading', {
            detail: { loading: true }
        }));

        // 2. Build Query Parameters
        const params = new URLSearchParams();
        if (e.detail.search) params.append('search', e.detail.search);
        
        if (e.detail.sort) {
            params.append('sort_field', e.detail.sort.field);
            params.append('sort_dir', e.detail.sort.direction); // 'asc' or 'desc'
        }

        // Filters object: { fieldName: ['val1', 'val2'] }
        if (e.detail.filters) {
            Object.keys(e.detail.filters).forEach(field => {
                const values = e.detail.filters[field];
                values.forEach(val => params.append(`filter_${field}[]`, val));
            });
        }

        // 3. Fetch from Server API
        fetch('/api/products?' + params.toString(), {
            headers: { 'Accept': 'application/json' }
        })
        .then(response => response.json())
        .then(res => {
            // 4. Set data and hide loading indicator
            tableEl.dispatchEvent(new CustomEvent('ln-table:set-data', {
                detail: {
                    data: res.data,          // Array of records
                    total: res.total,        // Total count of unfiltered records
                    filtered: res.filtered,  // Total count of search/filtered matches
                    filterOptions: res.options // Optional: preset filter choices
                }
            }));
        })
        .catch(err => {
            console.error('Failed to load table data:', err);
            tableEl.dispatchEvent(new CustomEvent('ln-table:set-loading', {
                detail: { loading: false }
            }));
        });
    });
})();

2. In-Memory Client-Side Coordinator

For small datasets preloaded on the page. Sorts, searches, and filters the array inside the browser instantly without network requests.

(function () {
    const tableEl = document.getElementById('products-table');
    const ALL_ITEMS = window.__INITIAL_DATA__ || []; // Your source array
    if (!tableEl) return;

    tableEl.addEventListener('ln-table:request-data', function (e) {
        if (e.detail.table !== 'products') return;

        let result = ALL_ITEMS.slice();

        // 1. Apply Search Filter
        if (e.detail.search) {
            const query = e.detail.search.toLowerCase();
            result = result.filter(item => 
                item.name.toLowerCase().includes(query) || 
                item.category.toLowerCase().includes(query)
            );
        }

        // 2. Apply Column Filters (AND logic across fields, OR within a field)
        if (e.detail.filters && Object.keys(e.detail.filters).length > 0) {
            const activeFilterFields = Object.keys(e.detail.filters);
            result = result.filter(item => {
                return activeFilterFields.every(field => {
                    const values = e.detail.filters[field];
                    if (values.length === 0) return true;
                    // Check if item field matches any of selected filter values
                    const itemVal = (item[field] || '').toString().toLowerCase();
                    return values.some(v => v.toLowerCase() === itemVal);
                });
            });
        }

        // 3. Apply Column Sorting
        if (e.detail.sort) {
            const field = e.detail.sort.field;
            const multiplier = e.detail.sort.direction === 'desc' ? -1 : 1;
            result.sort((a, b) => {
                const valA = a[field] || '';
                const valB = b[field] || '';
                if (typeof valA === 'number' && typeof valB === 'number') {
                    return (valA - valB) * multiplier;
                }
                return valA.toString().localeCompare(valB.toString()) * multiplier;
            });
        }

        // 4. Return Data
        tableEl.dispatchEvent(new CustomEvent('ln-table:set-data', {
            detail: {
                data: result,
                total: ALL_ITEMS.length,
                filtered: result.length
            }
        }));
    });
})();

Common mistakes

Putting row markup directly in <tbody>

<!-- WRONG -->
<tbody data-ln-data-table-body>
    <tr data-ln-row>
        <td>{{ title }}</td>
    </tr>
</tbody>

The component clears <tbody> on every set-data and replaces it with clones from the row template. Static rows are wiped on the first render. Put the row layout inside <template data-ln-template="{name}-row">.

Single sort icon with JS-driven <use href> swap

<!-- WRONG -->
<button data-ln-col-sort>
    <svg class="ln-icon"><use href="#ln-arrows-sort"></use></svg>
</button>

Tempting because "one icon, JS swaps the href on sort." It works visually but the wrong layer is doing the work — JS is mutating SVG attributes to convey state that already lives in a class on the <th>. The library shape is three sibling icons; CSS shows the right one. Use the three-icon pattern.

Building rows in JS instead of using the template

// WRONG
data.forEach(r => {
    tbody.innerHTML += `<tr><td>${r.title}</td></tr>`;
});

Bypasses the template, opens an XSS hole on r.title, and pulls row layout into the script file. Use set-data; let the component render via the template.
