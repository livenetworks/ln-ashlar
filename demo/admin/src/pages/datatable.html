Overview

Event-driven data table. Receives data via ln-data-table:set-data, renders rows from <template>. Sort, filter, search, selection, virtual scroll, keyboard navigation.

The table is a render engine, not a data source. It announces what it needs through ln-data-table:request-data and waits for a coordinator to feed records back via ln-data-table:set-data. Server-driven, client-cached or static — the table doesn't care; it only knows about markup, events and rendering.

This page documents the markup anatomy, then exercises every feature on a 10,000-row demo. For the full guide see the ln-data-table README and the architecture reference.

Anatomy

The minimal shape — every working table is a variation on this skeleton.

<section data-ln-data-table="documents" data-ln-data-table-selectable>

    <header> <!-- toolbar (yours) -->
        <input type="search" data-ln-data-table-search>
    </header>

    <table>
        <thead>
            <tr>
                <th data-ln-col-select></th>
                <th data-ln-col="title">
                    Title
                    <button data-ln-col-sort aria-label="Sort">
                        <svg data-ln-sort-icon="none"><use href="#ln-arrows-sort"></use></svg>
                        <svg data-ln-sort-icon="asc"> <use href="#ln-arrow-up"></use></svg>
                        <svg data-ln-sort-icon="desc"><use href="#ln-arrow-down"></use></svg>
                    </button>
                </th>
                <th data-ln-col="department">
                    Department
                    <button data-ln-col-filter><svg><use href="#ln-filter"></use></svg></button>
                </th>
            </tr>
        </thead>
        <tbody data-ln-data-table-body></tbody>
    </table>

    <template data-ln-template="documents-row">
        <tr data-ln-row>
            <td><input type="checkbox" data-ln-row-select></td>
            <td data-ln-cell="title"></td>
            <td data-ln-cell="department"></td>
        </tr>
    </template>
</section>

The component root — <section data-ln-data-table="...">

The attribute value is the table name, unique per page. It's echoed in every event the table dispatches (detail.table === 'documents') so a coordinator listening at document level can route by name. data-ln-data-table-selectable is opt-in — it switches on row checkboxes, select-all, and the ln-data-table:select event.

The toolbar — yours, the table just listens

The component does not build the toolbar. It only watches for one element: an <input> carrying data-ln-data-table-search. Each keystroke updates state, dispatches ln-data-table:search and triggers a fresh request-data. Place the input wherever it fits the page layout.

Column headers — <th data-ln-col="field">

The string after data-ln-col= is the data field name. It ties the column to the row template — <th data-ln-col="title"> matches <td data-ln-cell="title">. Without data-ln-col, the column has no field to sort or filter on; it's treated as presentational (the "Actions" column is the typical example).

The sort button — three icons, one source of truth

The sort button contains three sibling <svg> icons, distinguished by data-ln-sort-icon="none|asc|desc". All three sit in the DOM at all times. CSS shows exactly one based on a class on the ancestor <th>:

  • Default (no class) → the "none" icon (ln-arrows-sort) is visible.
  • <th class="ln-sort-asc"> → the "asc" icon (ln-arrow-up) is visible.
  • <th class="ln-sort-desc"> → the "desc" icon (ln-arrow-down) is visible.

The component never swaps an SVG, never replaces an icon URL, never sets inline styles. It toggles a class on the <th>; CSS does the rest. This is the cleanest example in the library of markup-driven state — every visual change is a class flip, and the icons live declaratively in HTML where you can read them.

The row template — <template data-ln-template="{name}-row">

The component clones this template once per record and fills cells: data-ln-cell="field" sets textContent, data-ln-cell-attr="field:attr" sets attributes (comma-separated for multiple). Why a <template> and not a static row? A static row would render empty in the initial paint. A <template> is inert until cloned — the component owns when rows appear. Why not build rows in JS? Because then the row layout lives in your script file and you'd have to redeploy JS to add a column.

Cell text is set via textContent, never innerHTML

This is intentional and load-bearing — textContent is XSS-safe. Records can contain any string, including <script> tags or attribute-injection payloads, and they will render as visible text rather than be parsed as markup. If you need rich content per cell, render plain text and use row-click to open a detail view.

The footer — count spans with hide-on-zero wrappers

Three optional spans report numbers: data-ln-data-table-total, data-ln-data-table-filtered, data-ln-data-table-selected. The "filtered" and "selected" spans live inside -wrap parents that get the hidden class when the count is zero — so the punctuation around them ("· 12 filtered") disappears with the value. No awkward " · 0 filtered" or empty leading separator.

Events

All events bubble from the [data-ln-data-table] element. Listen at document level for cross-component coordination, or directly on the table for tight scoping. None of the events are cancelable — the table announces; the coordinator decides.

Emitted by table

EventWhendetail
ln-data-table:request-dataInit, sort, filter, search, clear{ table, sort, filters, search }
ln-data-table:renderedRows rendered into <tbody>{ table, total, visible }
ln-data-table:sortUser clicks sort{ table, field, direction }
ln-data-table:filterUser changes filter{ table, field, values }
ln-data-table:clear-filtersUser clears all filters{ table }
ln-data-table:searchUser types in search{ table, query }
ln-data-table:row-clickUser clicks a row{ table, id, record }
ln-data-table:row-actionUser clicks action button{ table, id, action, record }
ln-data-table:selectSelection changes{ table, selectedIds, count }

Common mistakes

Putting row markup directly in <tbody>

<!-- WRONG -->
<tbody data-ln-data-table-body>
    <tr data-ln-row>
        <td data-ln-cell="title"></td>
    </tr>
</tbody>

The component clears <tbody> on every set-data and replaces it with clones from the row template. Static rows are wiped on the first render. Put the row layout inside <template data-ln-template="{name}-row">.

Mismatched data-ln-col and data-ln-cell field names

<!-- WRONG — column says "title", cell says "name" -->
<th data-ln-col="title">Title</th>
...
<td data-ln-cell="name"></td>

The cell renders empty because the record has no name field, or the sort request sends field: "title" but the data was prepared under name. Both attributes are the same identifier — pick a convention (the API field name) and align both sides.

Single sort icon with JS-driven <use href> swap

<!-- WRONG -->
<button data-ln-col-sort>
    <svg class="ln-icon"><use href="#ln-arrows-sort"></use></svg>
</button>

Tempting because "one icon, JS swaps the href on sort." It works visually but the wrong layer is doing the work — JS is mutating SVG attributes to convey state that already lives in a class on the <th>. The library shape is three sibling icons; CSS shows the right one. Use the three-icon pattern.

Building rows in JS instead of using the template

// WRONG
data.forEach(r => {
    tbody.innerHTML += `<tr><td>${r.title}</td></tr>`;
});

Bypasses the template, opens an XSS hole on r.title, and pulls row layout into the script file. Use set-data; let the component render via the template.
