Overview

A 141-line localStorage-backed draft buffer for HTML forms. Set data-ln-autosave on a <form> with an id and the form's serialised state is written to localStorage on every focusout and change, restored on construction, and cleared on submit, reset, or any click on a [data-ln-autosave-clear] button. The user never sees it work and never sees it not work; they just notice that their form is still filled in when they come back.

Storage is localStorage, not IndexedDB or a server endpoint. localStorage is synchronous (so the save on focusout has no race window with submit), origin-isolated, persisted across tab close, and universally available. ln-store uses IndexedDB for record cache; ln-autosave uses localStorage for form drafts. Different layers, different lifecycles, different keys (ln-autosave: vs ln_app_cache).

The save trigger is focusout + change, not input. Field-boundary events are naturally rate-limited by user behaviour — one save per field-leave, not one per keystroke. There is no data-ln-autosave-debounce attribute. The trade-off: a user typing continuously into a single textarea for 30 minutes with no blur triggers ZERO saves. For compose-style editors this is the wrong tool; see "Mistake 5" in the README.

The clear trigger is three signals. Submit means "the draft has become a real record." Reset means "user explicitly threw the draft away." [data-ln-autosave-clear] on a non-submit non-reset button means "Cancel" — discard the draft without resetting form values. Three signals, one effect.

For the full guide see the ln-autosave README and the architecture reference.

Anatomy

The minimum invocation is one attribute on the form:

<form id="edit-user" data-ln-autosave>
    <input name="name" type="text">
    <input name="email" type="email">
    <button type="submit">Save</button>
</form>

The storage key is ln-autosave:{pathname}:{identifier} — unique per URL + form. The identifier is the value of data-ln-autosave if present, falling back to form.id. Empty attribute (data-ln-autosave="") falls back to form.id. Missing both → console.warn and the constructor exits without instantiating.

Why localStorage and not something else

localStorage is synchronous (no race window with submit), origin-isolated, persisted across tab close, and universally available. sessionStorage would lose the draft on tab close — that's the headline scenario. IndexedDB would make every save async and force the component to think about race conditions on submit. A server endpoint would couple "save the draft" to "the network is up," defeating the resilience guarantee. localStorage's ~5MB cap is the price; for typical forms this is generous.

What state lives where

"Should this form autosave?" lives on data-ln-autosave on the <form>. The currently-saved draft lives in localStorage["ln-autosave:{path}:{id}"]. "Discard the draft on click" lives on data-ln-autosave-clear on any descendant button. There is no instance-level state worth speaking of — no in-memory cached draft, no debounce timer, no in-flight save promise. The component is essentially a listener bundle.

The pathname and identifier are captured ONCE at construction. Mutating data-ln-autosave later, or pushing a new History API URL while the form remains in the DOM, does NOT re-key the instance. To switch keys, destroy and re-mount.

data-ln-autosave<form>authorMarks the form as autosaved. Empty value uses form.id as the storage identifier; non-empty value uses the attribute value instead. If neither resolves, the constructor logs a warning and exits.

There are no other attributes. No data-ln-autosave-debounce, no -trigger, no -expire. The contract is intentionally narrow.

Events

All events bubble. Only before-restore is cancelable. detail.target is always the form element.

EventCancelableDetailWhen
ln-autosave:before-restoreyes{ target, data }Inside restore(), after parsing localStorage, BEFORE applying values. Calling preventDefault() skips the apply step entirely; :restored does NOT fire afterwards.
ln-autosave:restoredno{ target, data }After populateForm() has written values AND after synthetic input + change have been dispatched on every restored field. ln-validate / ln-autoresize have already re-reacted by this point.
ln-autosave:savedno{ target, data }After localStorage.setItem succeeds. If the write throws (quota exceeded, localStorage disabled), the try/catch swallows silently and NO event fires — there is no :save-failed event.
ln-autosave:clearedno{ target }After localStorage.removeItem runs. Idempotent — fires even if there was nothing in localStorage to remove.
ln-autosave:destroyedno{ target }Last action of destroy(), after listeners are removed but before el.lnAutosave is deleted.

API — el.lnAutosave

						

							
The instance lives on the form element as form.lnAutosave. Methods:


const form = document.getElementById('edit-user');

form.lnAutosave.save();      // serialise the form, write to localStorage, dispatch :saved
form.lnAutosave.restore();   // re-read localStorage, dispatch :before-restore (cancelable),
                             // populate, dispatch synthetic input/change, dispatch :restored
form.lnAutosave.clear();     // remove from localStorage, dispatch :cleared
                             // (does NOT reset form values — only wipes localStorage)
form.lnAutosave.destroy();   // remove all listeners, dispatch :destroyed, delete back-reference
                             // (does NOT call clear() — destroy is not "discard the draft")

form.lnAutosave.key;         // the resolved storage key string
form.lnAutosave.dom;         // back-reference to the form

// Manual init scan (Shadow DOM, iframe). The shared MutationObserver
// covers everything inside document.body — call this only for contexts
// the observer cannot see.
window.lnAutosave(container);

							
There is no getDraft(), hasDraft(), or setKey(). To inspect a draft without applying it, read localStorage directly using form.lnAutosave.key.

						

					


					
					

						
Cancel restore — server data takes priority

						

							
The canonical edit-page pattern. The server has already rendered the form with current record values; a stale draft restoring over them produces inconsistent state.


<form id="edit-user" data-ln-autosave data-has-server-data="true" action="/users/42" method="POST">
    <input name="name" type="text" value="Maria"> <!-- server-rendered -->
    <input name="email" type="email" value="maria@example.com">
    <!-- ... -->
</form>


document.addEventListener('ln-autosave:before-restore', function (e) {
    const form = e.detail.target;
    if (form.dataset.hasServerData === 'true') {
        e.preventDefault();              // do not apply the draft
        form.lnAutosave.clear();         // and discard it permanently
    }
});

							
The data-has-server-data attribute is a project convention, not an ln-autosave feature. The listener inspects whatever signal the form carries — could be form.dataset.recordId, could be a hidden input, could be a non-empty value field — and decides case-by-case.

						

					


					
					

						
Modal forms — close-and-discard

						

							
Inside a modal, three exit paths typically need to wipe the draft: the X icon, the Cancel button, and the platform submit on Create. Combine data-ln-modal-close with data-ln-autosave-clear on the same button:


<div id="invoice-modal" class="ln-modal" data-ln-modal>
    <form id="invoice" data-ln-autosave action="/invoices" method="POST">
        <header>
            <h3>New Invoice</h3>
            <button type="button" aria-label="Close"
                    data-ln-modal-close data-ln-autosave-clear>
                <svg class="ln-icon" aria-hidden="true"><use href="#ln-x"></use></svg>
            </button>
        </header>
        <main>
            <input name="client" type="text" required>
            <input name="amount" type="number" required>
        </main>
        <footer>
            <button type="button" data-ln-modal-close data-ln-autosave-clear>Cancel</button>
            <button type="submit">Create</button>
        </footer>
    </form>
</div>

							
Note: if the user dismisses the modal via ESC (which ln-modal handles natively), the draft is NOT cleared — the ESC-close path does not fire submit, reset, or click any clear button. To wire ESC-close to discard, listen for the modal's close event and call form.lnAutosave.clear().

						

					


					
					

						
Cross-component reality

						

							
ln-autosave does NOT import, listen for, or coordinate with any other component's custom events. The platform's input, change, submit, reset are the only connectors. Compositions that "just work":

							

							

								
ComponentCoordination

								
									
ln-formNone at the JS level. Both share serializeForm / populateForm from ln-core/helpers.js. The form's submit event is the natural connector — ln-form catches it for AJAX submission, ln-autosave catches it to clear the draft. No imports.

									
ln-validateNone directly. After :restored, the synthetic input / change events bubble up; ln-validate's input listener catches them and validates. Post-restore validation is correct without coordinator wiring.

									
ln-autoresizeNone directly. Same propagation path — restored textareas grow to fit because the synthetic input reaches the autoresize listener.

									
ln-select (TomSelect)The el.lnSelect.setValue() branch in the source (lines 97-99) is dead code — TomSelect instances are stored in a module-scoped WeakMap, NOT as el.lnSelect. The integration works via the dispatched change event on the underlying <select> instead.

									
ln-storeNone. Different layer entirely. ln-store uses IndexedDB for record cache; ln-autosave uses localStorage for form drafts. See "Mistake 7" in the README.

									
ln-toastNone. ln-autosave is silent by default — no "draft saved" toasts. Listen for ln-autosave:saved and dispatch your own toast if wanted (usually noisier than an inline aria-live indicator).

									
ln-modalNone directly. A <form data-ln-autosave> inside a modal works without wiring. The Cancel button uses data-ln-modal-close + data-ln-autosave-clear.

									
ln-core/persist.jsIndependent storage layers. ln-core/persist uses ln: prefix for UI state; ln-autosave uses ln-autosave: for form drafts. The two cannot collide.

								
							

							

						

					


					
					

						
Common mistakes

						

							
    
								
  • Putting the attribute on a wrapper <div> instead of the <form>. The form's elements collection only exists on actual <form> elements, and submit / reset events do not fire on divs. Always put data-ln-autosave on the form itself.
    • 
								
  • Expecting clear() to reset the form. It removes the localStorage entry, period. Form values are untouched. To reset both, call form.reset() (which also fires reset, which calls clear() internally — they compose).
    • 
								
  • Saving sensitive fields. localStorage is plaintext, accessible to any script on the origin (XSS risk). Drop data-ln-autosave on sensitive forms entirely, or filter sensitive keys in a :saved listener (rewriting localStorage post-save).
    • 
								
  • Two forms with the same key on the same page. They share localStorage, race on save, and overwrite each other. Use distinct identifiers (data-ln-autosave="step1", data-ln-autosave="step2") or collapse into one form.
    • 
								
  • Long single-field forms (compose textareas) with no field-boundary saves. The user types for 30 minutes without blurring; the draft was never saved. Add a periodic save (setInterval(() => form.lnAutosave.save(), 60000)) or move to an input-based save in project code.
    • 
								
  • Form schema changes between save and restore. Renamed field names produce stranded localStorage data and empty fields. Migrate at restore time by mutating e.detail.data in the :before-restore listener, OR schema-version the key (data-ln-autosave="user-form-v2").
    • 
								
  • Confusing ln-autosave with ln-store. Form drafts → ln-autosave (localStorage, per-form, per-URL). Record cache → ln-store (IndexedDB, per-table, application-wide).
    • 
								
  • Expecting cross-tab sync. The storage event is not observed. Two tabs with the same form do not sync as edits happen — only on hard refresh. ln-autosave is for single-tab "my laptop closed" recovery, not collaborative editing.
    • 
								
  • Calling form.submit() programmatically and expecting the draft to clear. form.submit() bypasses the submit event entirely. Use form.requestSubmit() instead — it fires the platform validation AND the submit event, which ln-autosave catches.
    • 
								
  • Storage quota exceeded silently. A draft pushing past 5MB causes setItem to throw QuotaExceededError; ln-autosave's try/catch swallows it without any event or warning. There is no :save-failed event today (asymmetric with ln-store:quota-exceeded). For huge forms, monitor proactively.
