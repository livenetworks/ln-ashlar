Overview & Philosophy

Confirming dangerous or destructive actions (like deleting a user or clearing a database) is a critical requirement in modern admin interfaces. Traditionally, libraries solve this by blocking interaction with heavy modals or using browser-native window.confirm() dialogues, which freeze thread execution, look unstyled, and disrupt developer flows.

In ln-ashlar, this is solved by ln-confirm, an ultra-lightweight **Interaction Gate Primitive** (~109 lines of JavaScript). Instead of popping up disruptive separate overlays, it turns the target button itself into a self-contained, two-click confirmation gate. It swallows the first click, morphs in-place to reveal the confirmation prompt, and releases the second click directly to the platform's native event processors.

The In-Place Action Gate

The confirmation action IS the platform click. The script intercepts the first click via event prevention and propagation stopping, but releases the second click completely. If clicked within the armed timeout, the form submits, the link navigates, and existing AJAX handlers fire naturally. There is no rewrite of the default action.

Orthogonal Layers of Danger Gating

Destructive confirmation is achieved by separating visual morphs, automatic timing, and platform defaults:

  1. The Platform Default (Trigger Action): The core action — standard form submits (type="submit") or link targets (href) — is declared natively on the HTML button or link, untouched by the component.
  2. The Visual Morph (In-Place States): The button represents its state through HTML attributes (data-confirming="true"). CSS mixins (@include confirm-tooltip) read attributes to transition text values or reveal overlay icon tooltips.
  3. The State Gate (ln-confirm): The JavaScript component coordinates the armed state. It schedules the countdown timer, intercepts the first user interaction, and reverts labels or icon paths on expiration.

How the Interaction Flows

  1. The user clicks a destructive button. The component intercepts the event, calls e.preventDefault() and e.stopImmediatePropagation(), transitions the button to data-confirming="true", replaces its label/icon with the armed prompt, schedules an auto-revert timer, and dispatches ln-confirm:waiting.
  2. If the user clicks the armed button again before the timer expires, the component intercepts the second click, runs cleanups synchronously, and lets the event pass through. The browser executes the natural submit, navigate, or AJAX handler.
  3. If the timer expires before a second click arrives, the button automatically clears the armed state, restores its original text or icon, and disarms.

For the full guide see the ln-confirm README and the architecture reference.

Anatomy

The minimum invocation is one attribute on the destructive button:

<button data-ln-confirm="Are you sure?">Delete record</button>

Two render modes — text button and icon-only button

The component branches at confirm time on whether the button has any text content at construction (it captures dom.textContent.trim() in the constructor and reads it back later):

  • Text button (constructor saw non-empty text) — confirm-state replaces textContent with the prompt. The button visually says "Are you sure?" while armed and reverts to its original label on revert.
  • Icon-only button (constructor saw empty text) — confirm-state finds the inner svg.ln-icon use, swaps its href to #ln-check, adds .ln-confirm-tooltip on the button, and writes the prompt text into data-tooltip-text. CSS reads the attribute and renders an ::after tooltip bubble above the button. ln-icons watches the <use href> mutation and ensures the swap target icon is in the sprite.

The branch is determined ONCE at construction. If you mount data-ln-confirm on an empty-text button and later add text content, the next confirm cycle still treats it as an icon button (because this.originalText was captured empty on construction). Re-init by calling btn.lnConfirm.destroy() followed by re-creating the instance, or set the text BEFORE the attribute lands.

What state goes where

"Should this button confirm before action?" lives on data-ln-confirm (presence). "Currently armed?" lives on data-confirming="true" on the button — the public CSS hook. "Auto-revert timeout (seconds)" lives on data-ln-confirm-timeout. "Original label" / "original icon href" live on the JS instance, captured at construction (text) or at first arm (icon).

The underlying destructive action — type="submit", href, an existing click listener — is owned by you, untouched by ln-confirm. The component does not run it; the second click does, by way of the platform's default action handling.

Example — Text buttons

Click once: the button morphs into the prompt and turns red. Click within 3 seconds: the click passes through (these have no type="submit" and no onclick, so nothing happens — that's the point: the component does not cause an action, it gates an existing one). Wait 3 seconds: the button auto-reverts.

Example — Icon-only buttons

The constructor sees empty text and a svg.ln-icon use child, so the icon-only branch runs on first click: the trash icon swaps to a check icon, the button gets .ln-confirm-tooltip + data-tooltip-text="Delete?", and CSS renders a tooltip bubble above the button via @mixin confirm-tooltip. The icon-only branch also handles screen-reader output — it captures the original aria-label ("Delete"), swaps it to the prompt text ("Delete?"), and appends a transient <span class="sr-only" role="alert"> announcer inside the button. AT speaks the prompt on first click; on revert, the announcer is removed and the original aria-label is restored. Try it with VoiceOver / NVDA.

Example — Custom timeout

Use data-ln-confirm-timeout to override the default 3-second auto-revert. Read fresh at first-click time only — mutating the attribute mid-armed does NOT extend the running timer (the attribute is not in the observer's attributeFilter; see "Common mistakes" item 4 in the README).

Example — Inside a form (the canonical pattern)

The destructive flow that ln-confirm exists for. First click arms the button. Second click within timeout — _reset runs synchronously, the click continues through the platform's default action processor, the form's submit event fires (because nothing called preventDefault), and the form submits via its native action / method. No JS rewiring; type="submit" is the contract.

This demo intercepts the submit event with a toast just to keep the page from navigating — production code would let the platform POST run.

Example — Non-form button with AJAX click handler

When the destructive action is an AJAX call that does NOT go through a form, attach the destructive code to the button's click listener as you normally would. ln-confirm gates the first click via preventDefault + stopImmediatePropagation, so this listener does NOT run on the first click. On the second click, ln-confirm returns silently (no preventDefault), so this listener runs.

Listener attachment order matters. stopImmediatePropagation only blocks listeners on the SAME element that have not yet run. ln-confirm attaches at construction time (typically DOMContentLoaded or shortly after); listeners attached BEFORE then will fire on the first click too. In normal flow this is rarely an issue because ln-ashlar loads early.

Example — Listening for arm events

The one event ln-confirm dispatches: ln-confirm:waiting on the first click, AFTER the visual swap and timer schedule, with { target } in the detail. Use it for analytics ("user almost-deleted"), page-level chrome ("show a Cancel All button while any confirm is armed"), or debugging.

Note that by the time the event fires, the button's textContent is the prompt text ("Are you sure?"), not the original label. Read btn.lnConfirm.originalText if you want the original label, or attach a more specific listener that knows the button identity.

Arm event log will appear here.

Example — Multiple confirms in a button group

Each [data-ln-confirm] button is its own instance with its own arm-state. Arming one does NOT arm the others; arming one does NOT reset already-armed siblings. Each runs its own auto-revert timer.

Example — Inside [data-ln-table] (overflow escape)

ln-data-table uses overflow: clip on [data-ln-table], which would clip the icon-only mode's tooltip bubble. The co-located rule [data-ln-table]:has([data-confirming]) { overflow: visible; } reverts the table to overflow: visible while ANY descendant button is armed. The moment the button reverts, :has() unmatches and the clip is restored.

API

const btn = document.querySelector('[data-ln-confirm]');

btn.lnConfirm.confirming;       // boolean — true while armed
btn.lnConfirm.originalText;     // string — captured at construction
btn.lnConfirm.confirmText;      // string — captured at construction

btn.lnConfirm.destroy();        // _reset() if armed, then unbind click listener,
                                //   delete instance reference. data-ln-confirm
                                //   attribute stays in place — caller controls
                                //   markup.

window.lnConfirm(root);         // re-run init scan over `root`. Only needed for
                                //   Shadow DOM / iframe roots the global
                                //   MutationObserver cannot see.

There is no arm(), accept(), or cancel() method. The component shape is "click is the only input." Programmatic open/close would contradict the in-place model — the second click IS the accept, and there is nothing else for ln-confirm to do.

Common mistakes

1. Listening for an :accept event that does not exist

There is no ln-confirm:accept. The accept semantic IS the platform click on the second click — attach to the form's submit, the button's click, or the link's navigation. ln-confirm:waiting (the only event) fires on the FIRST click, not the second.

2. Both an onclick and a <form action>

Both run on the second click, in order — the in-place pattern doesn't change platform behavior. If only one should run, drop the other or call e.preventDefault() in the click handler.

3. Double-click protection is the consumer's job

Once the second click passes through, ln-confirm is done. If the user clicks again during the AJAX round-trip, the action runs again. Disable the button in your submit / click handler, or wire data-ln-form which handles aria-busy.

4. Expecting data-ln-confirm-timeout mutations to take effect mid-confirm

Does NOT work. The attribute is read fresh at first-click time only, and is not in the observer's attributeFilter. Change the attribute BEFORE the user arms (in markup, or in coordinator code that updates the attribute when destructiveness changes). Earlier docs claimed this was reactive — that was a documentation bug.

5. Using <a data-ln-confirm> for destructive actions

Works (the second click navigates), but destructive actions over GET are an antipattern — pre-fetchers, link-preview crawlers, and shared-link unfurlers can fire GET requests at the URL without any user interaction. Use a form-submit button for actually-destructive actions.

6. Calling .click() programmatically and expecting two-clicks behavior

Two synchronous .click() calls pass through the gate (first arms, second clears). ln-confirm doesn't check isTrusted — it's a UX gate, not a security gate. Server-side checks (CSRF, second-factor) are the right defense against programmatic destruction.
