Store

Overview & Philosophy

In traditional web applications, the network is treated as a blocking dependency. Clicking a save button displays a spinner while blocking the UI, waiting for a round-trip network payload. If the connection is unstable, operations timeout, data is lost, and the user is blocked.

In ln-ashlar, this is solved by ln-store, a highly-optimized, zero-dependency **Local-First Sync Layer**. It acts as a client-side database (backed by standard browser IndexedDB) that decouples user interactions from network transport speeds. The application renders data directly from local storage, handles writes optimistically in real-time, and synchronizes data with the server in the background using a lightweight delta protocol.

The Local-First Doctrine

A local-first store never forces the UI to await network requests. Write actions are committed locally to IndexedDB instantly and dispatched to the interface, while background queues (with exponential backoff and version conflict resolution) sync mutations with the server. The local IndexedDB cache is the single source of truth for the UI.

Three Orthogonal Layers of the Data Pipeline

  1. The Cache Storage (IndexedDB): A native, highly-performant browser database capable of storing megabytes of structured JSON data. It supports indexing, fast text-based search queries, and stays persisted fully across app restarts and offline sessions.
  2. The Sync Engine (Delta Protocol): A background service that communicates with server API endpoints. It tracks timestamps, fetches only records changed since the last sync (delta arrays), processes conflict resolutions, and maintains outbox write queues.
  3. The Presentation Layer (UI Elements): Data tables (ln-data-table) and select controls render data directly from local IndexedDB queries. They listen to bubbled store events (like ln-store:created, ln-store:updated) to update lists in real-time.

How the Data Flow Coherence Works

When a record is created or mutated by the user:

  1. The UI dispatches an event (e.g. ln-store:request-create) with the form payload.
  2. The component assigns a temporary ID, saves the record locally, and instantly dispatches ln-store:created. Sibling tables receive the event and display the new record immediately.
  3. In the background, the sync outbox pushes a POST request. If the server confirms, it responds with the permanent ID and version, triggering ln-store:confirmed. If it fails or conflicts, the store dispatches ln-store:reverted or ln-store:conflict, letting the UI rollback safely.

For a detailed breakdown of local-first patterns, see the Data Flow Architecture, the ln-store README, and the architecture reference.

    Sync Lifecycle
    
						
    
							
    Init:
  1. Open IndexedDB → read _meta
  2. Schema mismatch? → clear, full load
  3. Has cache? → emit ready (source: 'cache'), delta sync if stale
  4. No cache? → full load → emit loaded, ready (source: 'server')

Visibility change (tab returns):
  → delta sync all stores if stale

Delta sync:
  1. GET {endpoint}?since={last_synced_at}
  2. Upsert data[], delete deleted[]
  3. Emit synced
    
						
    
					
    

					
					
    
						
    Optimistic Mutation Flow
    
						
    
							
    Create:  insert temp → POST → confirmed | reverted
Update:  save prev, update → PUT → confirmed | conflict | reverted
Delete:  save, remove → DELETE → confirmed | reverted
Bulk:    save all, remove → DELETE /bulk-delete → confirmed | reverted
    
						
    
					
    

					
					
    
						
    API
    
						
    
							
    const storeEl = document.querySelector('[data-ln-store="documents"]');

// Queries (all return Promise)
storeEl.lnStore.getAll(options)        // → { data, total, filtered }
storeEl.lnStore.getById(id)            // → Object|null
storeEl.lnStore.count(filters)         // → Number
storeEl.lnStore.aggregate(field, fn)   // → Number (sum, avg, count)

// State
storeEl.lnStore.isLoaded               // Boolean
storeEl.lnStore.isSyncing              // Boolean
storeEl.lnStore.lastSyncedAt           // Number|null
storeEl.lnStore.totalCount             // Number

// Manual triggers
storeEl.lnStore.forceSync()            // delta sync
storeEl.lnStore.fullReload()           // clear + full reload
storeEl.lnStore.destroy()              // cleanup

// Global
window.lnStore.clearAll()              // wipe all stores (logout)
window.lnStore.init(el)                // manual init
    
						
    
					
    

					
					
    
						
    Query Options
    
						
    
							
    storeEl.lnStore.getAll({
    sort: { field: 'title', direction: 'asc' },
    filters: { status: ['approved', 'draft'] },
    search: 'ISO 27001',
    offset: 0,
    limit: 100
})
// → { data: [...], total: 1247, filtered: 45 }
