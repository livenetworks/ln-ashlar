Store

Overview & Philosophy

In traditional web applications, the network is treated as a blocking dependency. Clicking a save button displays a spinner while blocking the UI, waiting for a round-trip network payload. If the connection is unstable, operations timeout, data is lost, and the user is blocked.

In ln-ashlar, this is solved by ln-store, a highly-optimized, zero-dependency **Local-First Sync Layer**. It acts as a client-side database (backed by standard browser IndexedDB) that decouples user interactions from network transport speeds. The application renders data directly from local storage, handles writes optimistically in real-time, and synchronizes data with the server in the background using a lightweight delta protocol.

The Local-First Doctrine

A local-first store never forces the UI to await network requests. Write actions are committed locally to IndexedDB instantly and dispatched to the interface, while background queues (with exponential backoff and version conflict resolution) sync mutations with the server. The local IndexedDB cache is the single source of truth for the UI.

Three Orthogonal Layers of the Data Pipeline

  1. The Cache Storage (IndexedDB): A native, highly-performant browser database capable of storing megabytes of structured JSON data. It supports indexing, fast text-based search queries, and stays persisted fully across app restarts and offline sessions.
  2. The Sync Engine (Delta Protocol): A background service that communicates with server API endpoints. It tracks timestamps, fetches only records changed since the last sync (delta arrays), processes conflict resolutions, and maintains outbox write queues.
  3. The Presentation Layer (UI Elements): Data tables (ln-table) and select controls render data directly from local IndexedDB queries. They listen to bubbled store events (like ln-data-store:created, ln-data-store:updated) to update lists in real-time.

How the Data Flow Coherence Works

When a record is created or mutated by the user:

  1. The coordinator claims the form submit and dispatches a local write event (e.g. ln-data-store:request-create) with the form payload and a temporary ID.
  2. The store saves the record locally and instantly dispatches ln-data-store:created. Sibling views receive the event and display the new record immediately.
  3. In the background, the coordinator pushes the network request via the connector or outbox queue. On server confirmation, the coordinator dispatches an ordinary ln-data-store:request-update to rekey the temporary ID to the server ID (emitting ln-data-store:updated). On a 4xx create rejection, it dispatches ln-data-store:request-delete (emitting ln-data-store:deleted).

For a detailed breakdown of local-first patterns, see the Data Flow Architecture, the ln-data-store README, and the architecture reference.

    Sync Lifecycle
    
						
    
							
    Init:
  1. Open IndexedDB → read _meta
  2. Schema mismatch? → clear, full load
  3. Has cache? → emit ready (source: 'cache'), delta sync if stale
  4. No cache? → full load → emit loaded, ready (source: 'server')

Visibility change (tab returns):
  → delta sync all stores if stale

Delta sync:
  1. GET {endpoint}?since={last_synced_at}
  2. Upsert data[], delete deleted[]
  3. Emit synced
    
						
    
					
    

					
					
    
						
    Optimistic Mutation Flow
    
						
    
							
    Create:  insert temp → POST → confirmed | reverted
Update:  save prev, update → PUT → confirmed | conflict | reverted
Delete:  save, remove → DELETE → confirmed | reverted
Bulk:    save all, remove → DELETE /bulk-delete → confirmed | reverted
    
						
    
					
    

					
					
    
						
    API
    
						
    
							
    const storeEl = document.querySelector('[data-ln-data-store="documents"]');

// Queries (all return Promise)
storeEl.lnDataStore.getAll(options)        // → { data, total, filtered }
storeEl.lnDataStore.getById(id)            // → Object|null
storeEl.lnDataStore.count(filters)         // → Number
storeEl.lnDataStore.aggregate(field, fn)   // → Number (sum, avg, count)

// State
storeEl.lnDataStore.isLoaded               // Boolean
storeEl.lnDataStore.isSyncing              // Boolean
storeEl.lnDataStore.lastSyncedAt           // Number|null
storeEl.lnDataStore.totalCount             // Number

// Manual triggers
storeEl.lnDataStore.forceSync()            // delta sync
storeEl.lnDataStore.fullReload()           // clear + full reload
storeEl.lnDataStore.destroy()              // cleanup

// Global
window.lnDataStore.clearAll()              // wipe all stores (logout)
window.lnDataStore.init(el)                // manual init

// → { data: [...], total: 1247, filtered: 45 }

    Reconnect auto-resync
    
	
    
		
    When the browser goes back online, ln-data-store automatically triggers a remote sync for all loaded stores — default on, no wiring required. The isSyncing guard makes it idempotent alongside manual forceSync() calls.
    
		
    Opt out per store: <div data-ln-data-store="chat" data-ln-data-store-no-autosync></div>
    
		
    Connectivity indicator (listens to ln-data-store:online / ln-data-store:offline on document):
    
		
    Online — resync triggered.
    
		
    Offline — cached data still available.
    
		
    Toggle your network connection to see the indicator update.
