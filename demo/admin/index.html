    ashlar-gui v1.2.0

    Philosophy

    An ashlar library.

    An ashlar is a finely cut block of stone — small, regular, ready to assemble. Cathedrals are built from ashlars. So are walls.

    ln-ashlar gives you the blocks. Tokens, mixins, components — small primitives, well-formed, predictable. You stack them into the building you actually need.

    How

    Token-driven. Spacing, color, radius, shadow — every value lives in --size-*, --color-*, --radius-*. Change one token, every consumer reacts. No hardcoded literals in components.

    Recipe + application. Each pattern has two layers: a @mixin (the recipe) and a default selector (the application). Use the default, apply the recipe to your own semantic selector, or rebind the surface tokens. Three first-class paths.

    Zero dependencies. Vanilla JS IIFE components, CustomEvent communication, MutationObserver auto-init. No framework, no runtime — drop the SCSS and JS into any project.

    Themes are palette only. A theme rebinds vocabulary tokens at its own :root. It never restructures a component. Switching themes is a token swap, not a rewrite.

    Why

    A library should give you primitives and get out of the way. Most CSS frameworks give you a finished building — beautiful for one project, awkward for the next.

    ln-ashlar gives you the stones. Stack them however your project needs. The stones are simple. The buildings can be as complicated as you want.

      ln-ajax — AJAX Navigation

      Component for AJAX navigation and form submission. Links and forms inside a data-ln-ajax container are automatically processed via fetch.

      AttributeDescription
      data-ln-ajaxContainer — all <a> and <form> inside are AJAX
      data-ln-ajax="false"Exclude a specific element from AJAX processing

      CSRF token automatically from <meta name="csrf-token">. History pushState for links and GET forms.
