    ashlar-gui v1.2.0

    Overview

    Per-field validation primitive for the form layer. 160 lines of JS that wrap the native ValidityState API, expose its state through two CSS classes and a pair of events, and add one strictly-typed escape hatch (set-custom / clear-custom) for errors the browser cannot express.

    The component does not run validation rules — the browser already does that. required, type="email", minlength, pattern, min, max are all native HTML attributes; dom.checkValidity() returns the boolean. ln-validate decides when to listen (input vs change, depending on field type), when to render (silent until the field has been touched at least once), and how to surface the result (CSS classes, .hidden toggles on error <li> elements, bubbling :valid / :invalid events).

    For the full guide see the ln-validate README and the architecture reference. The ln-form README is the companion read — the two components together are the form layer's contract.

    Anatomy

    The minimum invocation is data-ln-validate on a field plus a sibling <ul data-ln-validate-errors> error list inside a .form-element wrapper.

    <div class="form-element">
    <label for="email">Email</label>
    <input id="email" name="email" type="email" required minlength="5" data-ln-validate>
    <ul data-ln-validate-errors>
        <li class="hidden" data-ln-validate-error="required">Email is required</li>
        <li class="hidden" data-ln-validate-error="typeMismatch">Invalid email format</li>
        <li class="hidden" data-ln-validate-error="tooShort">Must be at least 5 characters</li>
    </ul>
</div>

    The field — [data-ln-validate]

    Marker on <input>, <select>, or <textarea>. Presence creates the instance; the value is ignored. Validation rules come from native HTML attributes — data-ln-validate is a participation marker, not a rule definition.

    The error list — [data-ln-validate-errors]

    A sibling <ul> inside the same .form-element wrapper. The component finds it via dom.closest('.form-element').querySelector('[data-ln-validate-errors]'), so the wrapper class must be .form-element — that string is hardcoded.

    The error message — [data-ln-validate-error="key"]

    Each <li> carries a key matching either a ValidityState property (mapped) or a custom-error name (unmapped). Both flavours live in the same <ul> — the component only renders native errors via the validation loop and only renders custom errors via the set-custom / clear-custom event handlers. The split is invisible in the markup; it only matters in the code path.

    What state goes where

    The instance state is two fields: _touched (Boolean — has the user interacted yet?) and _customErrors (Set of active custom-error names). The DOM holds the value; the browser holds validity. The CSS classes (ln-validate-valid, ln-validate-invalid) on the field are the visible state for project SCSS to hook. The .hidden class on each <li> is the per-message visibility toggle.

    The _touched gate — silent until first interaction

    The single most consequential UX decision in the component. Validation runs silently until the field has fired its first input or change event. Without this, every page load would render a wall of "this field is required" messages on every empty form. With it, errors appear only after the user has actually engaged with the field.

    The flag is forward-only during normal flow — once true, stays true until reset(). Note that validate() itself does NOT check _touched; force-validating an untouched field renders errors immediately. That is intentional — lnForm.submit() calls instance.validate() on every field exactly to catch untouched required fields and prevent submission.

    The flag is read by ln-form._updateSubmitButton, which disables the submit button if NO field has been touched yet (in addition to the standard "any field invalid" check). This implements "you can't submit a form you haven't started yet."

    The listener discriminator — input vs change

    The constructor reads the field's tag and type to decide which events to listen for:

    const isChangeBased = tag === 'SELECT' || type === 'checkbox' || type === 'radio';
    isChangeBasedField typesListeners
    falsetext, email, tel, url, password, number, search, date, time inputs; textareainput AND change
    true<select>, <input type="checkbox">, <input type="radio">change only

    The fork exists because input does not fire on SELECT, checkbox, or radio elements — the platform's contract for those is "value commits on change." This same discriminator is mirrored exactly in ln-form.fill(), ln-form.reset(), and ln-autosave's synthetic-event dispatch — programmatic value writes route through the same listener as user interaction.

    Programmatic API

    Force-validate every field above, reset all of them, or check current validity. Each button below operates on every [data-ln-validate] on this page.

    const input = document.getElementById('email');

input.lnValidate.validate();   // Force-validate now — returns Boolean
input.lnValidate.reset();      // Clear _touched, hide all errors, remove classes
input.lnValidate.isValid;      // Live getter — checkValidity() && _customErrors.size === 0
input.lnValidate.destroy();    // Detach listeners, dispatch :destroyed

    Attributes reference

    AttributeOn elementDescription
    data-ln-validate<input>, <select>, <textarea>Marks the field as participating. Creates the instance, attaches listeners. Value ignored.
    data-ln-validate-errors<ul>Identifies the error-message list. Must be inside the same .form-element wrapper as the field.
    data-ln-validate-error="key"<li>Identifies a specific error message. Key matches a ValidityState property (native, mapped) or a custom-error name (unmapped).
    .hidden (class)<li>Toggle target. Default-applied; removed by validate() (or set-custom) when the matching error is active.
    .ln-validate-valid (class)field (auto)Written by ln-validate when isValid is true.
    .ln-validate-invalid (class)field (auto)Written by ln-validate when isValid is false.

    Events — emitted

    EventCancelabledetailDispatched when
    ln-validate:validno{ target, field }After validate() determines the field is valid (native + no custom errors).
    ln-validate:invalidno{ target, field }After validate() determines the field is invalid.
    ln-validate:destroyedno{ target }After destroy() detaches listeners and removes the instance reference.

    All events bubble. detail.field is dom.name — the value of the name attribute, exposed for consumers that want to key per-field state without re-querying the DOM.

    Events — received

    EventdetailBehavior
    ln-validate:set-custom{ error: String }Adds error to _customErrors, sets _touched = true, unhides the matching <li>, flips CSS classes to invalid. Does NOT call validate() or dispatch :invalid.
    ln-validate:clear-custom{ error: String } or {}If error provided: removes from set, hides matching <li>. If empty: clears all custom errors, hides every custom <li>. Then if _touched, calls validate() to re-render native state.

    The asymmetry is intentional. set-custom is silent because the caller already knows the field is invalid. clear-custom must call validate() because removing one custom error might leave other customs active, native errors active, or none of the above — only a fresh validate() can resolve the new state.

    Cross-component coordination
    
						
    
							
    
							
    
								
    ComponentCoordination
    
								
									
    ln-formListens for bubbled :valid / :invalid, re-evaluates submit-button state by re-querying [data-ln-validate] fields and calling checkValidity() on each. On lnForm.submit(), force-validates every field. On lnForm.reset(), dispatches synthetic input/change first, then calls instance.reset() on every validate field — order matters.
    
									
    ln-autosaveNone directly. Restores values via populateForm(), dispatches synthetic input/change. ln-validate's listeners catch them and validate. Pure platform event flow.
    
									
    ln-confirmIndependent. Confirm gates the click on a destructive button; validate operates on form fields. Different elements, different event channels.
    
									
    ln-storeNone directly. ln-store dispatches ln-form:error on the form for 4xx server responses. ln-validate does NOT subscribe — the application or a future form-coordinator translates server errors into ln-validate:set-custom dispatches. The injection point is set-custom; the producer is consumer code.
    
									
    ln-autoresizeIndirectly. Both listen to input on textareas. They don't coordinate — they happen to react to the same trigger. A <textarea data-ln-validate data-ln-autoresize> resizes and validates simultaneously on every keystroke.
    
								    
							
    
							
    
							
    The pattern is consistent: ln-validate writes to the platform (classes, .hidden toggles, custom events) and reads from the platform (input, change, browser validity). Other components compose by doing the same. None of them import each other.
    
						
    
					
    

					
					
    
						
    Common mistakes
    
						
    
							
    1. Skipping the .form-element wrapper
    
							
    validate() uses dom.closest('.form-element') — null returns short-circuit the error-list lookup. CSS classes on the field still toggle, but no message renders.
    

							
    2. Programmatic el.value = '...' writes don't validate
    
							
    Setting .value is silent — neither input nor change fires. Either dispatch the event yourself (mirror the isChangeBased fork) or use lnForm.fill(data) which does the dispatch loop.
    

							
    3. Listening for :invalid after set-custom
    
							
    _onSetCustom is intentionally silent at the event level — visible-state changes only, no :invalid dispatch. To make ln-form's submit-button gate learn about custom errors, also call instance.validate() after the dispatch — that path dispatches :invalid, which triggers ln-form to re-evaluate.
    

							
    4. Relying on the browser's :invalid pseudo-class
    
							
    CSS :invalid fires on every empty required field on page load — exactly what _touched exists to avoid. Use .ln-validate-invalid instead.
    

							
    5. Putting data-ln-validate on a non-form element
    
							
    <div data-ln-validate> crashes on the first event — div.checkValidity() is undefined. Mark only inputs, selects, and textareas.
    

							
    6. Multiple <li>s with the same key
    
							
    Both unhide simultaneously — the user sees duplicate messages. Pick one and use a custom-error key for the variant.
    

							
    For the full list with code samples and explanations, see the README §Common mistakes.
