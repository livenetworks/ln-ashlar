# v1.1 Consumer Impact — Internal Cheat Sheet

> This is NOT a public migration guide. It's a cheat sheet for whoever
> (human or Claude) refactors a consumer project (DocuFlow, gls) from
> a pre-v1.1 ln-acme snapshot to v1.1.0.
>
> Purpose: when opening a fresh session without phase context loaded,
> read this file first to know what actually changed, what silently
> got better, and what grep patterns find stale code in the consumer.

---

## 1. Breaking changes — MUST address

These are patterns that exist in pre-v1.1 consumers and will either
break compilation, render wrong, or regress accessibility after the
ln-acme bump. Each has a grep to locate offenders.

### 1.1 Hardcoded light-mode colors

**Why:** v1.1 shipped full dark mode. Semantic tokens auto-switch on
`[data-theme="dark"]` and `@media (prefers-color-scheme: dark)`.
Consumers using hardcoded whites/greys won't switch — they become
stuck islands of light on a dark page.

**Grep the consumer for:**
```
background:\s*(white|#fff|#ffffff|hsl\(0 0% 100%\))
color:\s*(black|#000|#000000)
background:\s*hsl\(220\s+20%\s+98%\)     // hardcoded neutral-50
border-color:\s*hsl\(220\s+13%\s+91%\)   // hardcoded neutral-200
```

**Replace with semantic tokens:**
```scss
background: var(--color-bg);        // page card / form bg
background: var(--bg-elevated);     // subtle nested bg
background: var(--bg-recessed);     // page body bg
color:      var(--color-fg);        // headings, body
color:      var(--fg-muted);        // muted labels
color:      var(--fg-subtle);       // hints, placeholders
border:     1px solid var(--border-strong);   // standard edge
border:     1px solid var(--border-subtle);   // subtler edge
```

All of these invert correctly in dark mode without consumer action.

### 1.2 Raw animations without motion-safe gate

**Why:** v1.1 Phase 5 wrapped every stock animation in
`@include motion-safe { ... }` which gates on
`prefers-reduced-motion: no-preference`. Consumer-side animations
that aren't wrapped will still run for users who've opted out —
a regression against the rest of the library.

**Grep the consumer for:**
```
@keyframes
animation:
transition:\s*(transform|scale|translate|rotate|opacity)
```

**Rule of thumb:**
- Color/bg-color transitions on hover → **leave unwrapped**, they're fine
- Anything moving (`transform`, `translate`, `scale`, `rotate`) or
  any `@keyframes` → **wrap in `@include motion-safe`**

```scss
// WRONG (v1.0 pattern)
.card:hover { transform: translateY(-2px); }

// RIGHT (v1.1 pattern)
.card:hover {
    @include motion-safe {
        transform: translateY(-2px);
    }
}
```

### 1.3 Hand-rolled focus rings

**Why:** v1.1 Phase 4 shipped a three-layer focus halo via
`@include focus-ring`. Consumers that hand-rolled outlines or
box-shadows get weaker accessibility signal and look inconsistent.

**Grep the consumer for:**
```
:focus-visible\s*\{[^}]*outline
:focus-visible\s*\{[^}]*box-shadow
:focus\s*\{[^}]*outline
```

**Replace with one of six presets (from `scss/config/mixins/_focus.scss`):**
```scss
input:focus-visible  { @include focus-ring; }             // default
input:focus-visible  { @include focus-border-thicken; }   // structural
input:focus-visible  { @include focus-combination; }      // max signal
input:focus-visible  { @include focus-background-shift; } // minimal
input:focus-visible  { @include focus-accent-line; }      // subtle
input:focus-visible  { @include focus-inset-shadow; }     // tactile

// Error field:
.error:focus-visible { @include focus-ring(var(--color-error)); }
```

### 1.4 Hardcoded shadows

**Why:** v1.1 Phase 4 re-tuned shadows to cool-tinted
`hsl(220 40% 15% / alpha)` in light mode, and pure black in dark mode.
Hardcoded `rgba(0,0,0,0.1)` shadows look grey-on-grey in dark mode
and clash with the tinted v1.1 palette in light mode.

**Grep the consumer for:**
```
box-shadow:.*rgba\(0,\s*0,\s*0
box-shadow:.*#[0-9a-f]{3,6}
```

**Replace with tokens:**
```scss
box-shadow: var(--shadow-xs);   // hairline
box-shadow: var(--shadow-sm);   // card default
box-shadow: var(--shadow-md);   // hover / elevated card
box-shadow: var(--shadow-lg);   // dropdown, popover
box-shadow: var(--shadow-xl);   // modal
box-shadow: var(--shadow-2xl);  // max elevation
box-shadow: var(--shadow-inner); // pressed / sunk

// Color-aware (for CTAs and focus halos):
box-shadow: var(--shadow-primary);
box-shadow: var(--shadow-success);
box-shadow: var(--shadow-error);
```

## 2. Non-breaking but recommended

These compile fine but aren't using the new v1.1 API. Consumer works,
just doesn't benefit from the new ergonomics.

### 2.1 Manual font-size / line-height pairs → `typography($role)`

**v1.1 shipped role tokens.** Old per-size mixins (`@include text-sm`,
`@include font-semibold`) still exist and still work — they weren't
removed. But new code and refactor targets should use the role mixin:

```scss
// Old (still works)
h2 { @include text-2xl; @include font-bold; line-height: 2rem; }

// New (v1.1 canonical)
h2 { @include typography(heading-lg); @include font-bold; }
```

**Role catalogue** (from `scss/config/mixins/_typography.scss`):
```
display-lg  display-md  display-sm    // hero / landing
heading-lg  heading-md  heading-sm    // page / section titles
title-md    title-sm                   // card titles, subheadings
body-lg     body-md     body-sm        // paragraph text
label-md    label-sm                   // form labels, small UI
caption                                 // helper text, footnotes
```

Role mixin sets both `font-size` and `line-height` in one call by
reading the matching `--text-{role}` and `--lh-{role}` tokens.

### 2.2 Mixin param pattern — literal values, not tokens

**v1.1 convention:** spacing mixins (`p`, `px`, `py`, `gap`, `m`, etc.)
take literal values directly. Tokens are for CSS custom properties
(theming, runtime overrides), not mixin params.

```scss
// RIGHT (v1.1)
@include p(1rem);
@include px(1.5rem);
@include gap(0.75rem);

// WRONG (v1.0 pattern, still compiles but not the convention)
@include p(var(--size-md));
@include px(var(--size-lg));
```

Not a compilation error — just drift in consumer style. Update
opportunistically during the refactor, not as a forced grep pass.

## 3. Silent improvements — no consumer action needed

These get better automatically after the ln-acme bump. Consumer code
stays the same; behavior upgrades.

| Improvement | Where | How it activates |
|---|---|---|
| **Dark mode** | All library components | Consumer opts in by adding `[data-theme="dark"]` on `<html>` or by user's OS preference. Semantic tokens do the rest. |
| **Motion safety** | All library animations | Consumer's users who set `prefers-reduced-motion: reduce` get stock animations disabled. No code change in consumer. |
| **Cool-tinted shadows** | All shadow tokens | Already applied via `var(--shadow-*)`. Consumer that uses the tokens gets the new values for free. |
| **Focus halos** | Library components | Any consumer using default selectors (`<button>`, `<input>`, `[data-ln-modal]`) auto-gets the new focus ring. |
| **Neutral scale** | Semantic tokens | All `--color-bg-*`, `--color-text-*`, `--color-border*` reference the new neutral scale. Consumer that uses semantic tokens adapts automatically. |

Adopt one when the consumer has a matching custom component you want
to replace.

## 5. Foundation additions — optional adoption

### Breakpoints scale (Phase 1)
`scss/config/_breakpoints.scss` — `$bp-sm`, `$bp-md`, `$bp-lg`, `$bp-xl`, `$bp-2xl`.
Use for shell-level `@media` queries. Component-level responsive still
uses `@container`.

### Container query tokens (Phase 1)
`$cq-compact`, `$cq-medium`, `$cq-wide` for standardized component breakpoints.

### Max-width tokens (Phase 1)
`--max-w-prose` (65ch), `--max-w-form`, `--max-w-content`, `--max-w-container`.

### Easing tokens (Phase 1)
`--easing-standard`, `--easing-decelerate`, `--easing-accelerate`, `--easing-spring`.

## 6. Refactor workflow recommendation

Don't big-bang. Per-module order, visual check after each:

1. **Bump `ln-acme` dependency in consumer** (package.json → v1.1.0)
2. **Run `npm install` + dev server**
3. **Visual walk-through** of key pages. Note what looks wrong.
4. **Module-by-module** fixes in order of user-visibility:
   - Auth (login, magic link)
   - Shell (header, sidebar, layout)
   - Dashboard
   - Documents list + detail
   - Settings
   - Admin
5. **One commit per module** once its breakage is resolved.
6. **Final pass:** opportunistic upgrades (typography roles, chip for labels, stat-card for KPIs, etc.) as separate commits.

Safety net: if a module goes sideways, `git checkout` the consumer's
`package.json` to the pre-bump commit, then debug offline.

## 7. What this doc does NOT cover

- **JS components** — v1.1 was CSS-first. No JS component APIs changed.
  ln-popover, ln-tooltip (enhance), ln-copy, ln-shortcuts are v1.2
  scope and not shipped in v1.1.0.
- **Density system** — deferred to v1.2.
- **v1.0 → v0.x jumps** — if a consumer is on something pre-v1.0,
  handle that separately before attempting the v1.1 refactor.
