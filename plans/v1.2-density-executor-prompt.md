# Executor Prompt — v1.2 Scope B: Density System

> Single self-contained prompt for the executor. Implements the
> `[data-density="compact"]` global spacing compression system in
> ln-acme v1.2. SCSS-only. Touches no JavaScript.

---

## Context

ln-acme v1.2 introduces a **density system**: a single attribute
(`data-density="compact"`) flips the entire UI from comfortable to
dense spacing. Tables show more rows, forms pack tighter, cards shrink
padding. Same components, same markup, same selectors — only the
internal spacing tokens change.

This is implemented as a **CSS variable override**, not a component
override. New `--density-*` tokens default to comfortable values; under
`[data-density="compact"]` they re-tune to dense values. Components
that opt in reference these tokens instead of their hardcoded spacing.

Components that opt out (buttons, modals, page header, toast, nav)
keep their original tokens and stay untouched.

**Important codebase fact** — most `scss/components/_*.scss` files in
ln-acme are thin wrappers that just call `@include <name>` on a
default selector. The actual styling rules live in
`scss/config/mixins/_<name>.scss`. The plan lists the components
files as "retrofit targets," but the retrofit MUST happen in the
mixin files for the changes to actually take effect for all consumers
(both the default selector AND consumers who write
`#my-thing { @include card; }`). This prompt instructs you to edit
the mixin files. The components files stay untouched.

## Step 1 — Create `scss/config/_density.scss`

Create a new file at `c:\laragon\www\ln-acme\scss\config\_density.scss`
with these exact contents:

```scss
// ==========================================================================
// Density — ln-acme v1.2
// ==========================================================================
// Global spacing compression toggle. Flip one attribute, the entire
// UI tightens. Tables show more rows, forms pack denser, cards shrink
// padding. Same components, same markup, same selectors — only the
// internal spacing tokens change.
//
// Activation:
//   <html data-density="compact">         ← global
//   <main data-density="compact">         ← scoped to a region
//
// Density is an explicit user choice (consumer's settings toggle).
// It is NEVER auto-switched by viewport — viewport breakpoints handle
// layout, density handles information density inside that layout.
//
// What reacts: tables, forms, cards, stat-card.
// What does NOT react: buttons (a11y hit target), modal, toast,
// page header, sidebar nav.
//
// See docs/css/density.md for the full spec.
// ==========================================================================

:root {

	// -----------------------------------------------------------------------
	// Density — Padding scale
	// Mirrors --spacing-* but routed through density tokens so the
	// compact override below can shrink each step independently.
	// -----------------------------------------------------------------------
	--density-pad-xs:  0.25rem;   //  4px — tight inline padding (badges)
	--density-pad-sm:  0.5rem;    //  8px — table cells, form field py
	--density-pad-md:  1rem;      // 16px — cards, panel headers, form field px
	--density-pad-lg:  1.5rem;    // 24px — stat card, large panel padding

	// -----------------------------------------------------------------------
	// Density — Gap scale
	// For grid/flex gaps inside dense components.
	// -----------------------------------------------------------------------
	--density-gap-sm:  0.5rem;
	--density-gap-md:  0.75rem;
	--density-gap-lg:  1rem;

	// -----------------------------------------------------------------------
	// Density — Row height
	// Minimum row height for table rows and form inputs. Padding alone
	// determines visual height in comfortable mode; compact uses this
	// as a min-height floor so rows stay scannable when content is short.
	// -----------------------------------------------------------------------
	--density-row-h:    2.75rem;  // 44px — comfortable input/row min
	--density-row-h-sm: 2.25rem;  // 36px — compact-equivalent min for sub-elements

	// -----------------------------------------------------------------------
	// Density — Body typography
	// Controls only CONTENT text (table cell text, card body text,
	// stat trend line). Headings, labels, table headers, and section
	// titles do NOT shrink — they are structural and need to remain
	// scannable at any density.
	// -----------------------------------------------------------------------
	--density-font-body: var(--text-body-md);
	--density-lh-body:   var(--lh-body-md);
}

// ---------------------------------------------------------------------------
// Compact override
// Activated by [data-density="compact"]. Re-tunes every density token
// to its tighter value. The cascade does the rest — every component
// referencing these tokens shrinks automatically.
//
// Anti-pattern this avoids: writing
//   [data-density="compact"] [data-ln-card] { padding: ...; }
// for every component. That explodes specificity and duplicates
// rules. The token route keeps the override surface to one place.
// ---------------------------------------------------------------------------
[data-density="compact"] {
	--density-pad-xs:  0.125rem;
	--density-pad-sm:  0.375rem;
	--density-pad-md:  0.625rem;
	--density-pad-lg:  1rem;

	--density-gap-sm:  0.375rem;
	--density-gap-md:  0.5rem;
	--density-gap-lg:  0.75rem;

	--density-row-h:    2.25rem;
	--density-row-h-sm: 1.875rem;

	--density-font-body: var(--text-body-sm);
	--density-lh-body:   var(--lh-body-sm);
}
```

**Acceptance criterion:** file exists, `npm run build` (run later)
will not error on syntax.

## Step 2 — Wire into `scss/ln-acme.scss`

Density tokens must load **after `tokens`** (so `--text-body-md` etc.
are defined first) and **before `theme`** (so dark mode overrides
cascade on top of density without conflict — they are orthogonal axes
and dark mode never touches `--density-*`).

Open `c:\laragon\www\ln-acme\scss\ln-acme.scss`. Current top of file:

```scss
@use 'config/tokens';
@use 'config/breakpoints';
@use 'config/theme';
@use 'config/icons';
```

Insert the density `@use` between `breakpoints` and `theme`:

```scss
@use 'config/tokens';
@use 'config/breakpoints';
@use 'config/density';
@use 'config/theme';
@use 'config/icons';
```

**Why this exact position:**

- After `tokens` — density references `--text-body-md`, `--lh-body-md`, `--text-body-sm`, `--lh-body-sm` which are defined in tokens. CSS custom property ordering only matters at use-time, not declaration-time, but keeping the load order conceptually tidy makes the file self-documenting.
- Before `theme` — purely organizational. Density and dark mode are independent (no shared tokens). `theme.scss` only touches `--color-*` tokens; density only touches `--density-*` tokens. They cannot collide. Putting density before theme keeps the foundation block (tokens → density → theme) reading as "scale layer, density layer, color layer."

**Coordination note for the JS architect's parallel work:** the JS
architect (`feat(ln-popover): ...`) will add a co-located SCSS line
near the bottom of this file inside the "JS component co-located
styles" block. There is **no conflict** — the JS additions are far
below your single insertion at the top of the file. If you see a
merge conflict in this file when rebasing, it is a real conflict,
not a phantom — investigate.

**Acceptance criterion:** the line `@use 'config/density';` exists
between `@use 'config/breakpoints';` and `@use 'config/theme';`. No
other lines in this file are modified.

## Step 3 — Retrofit `scss/config/mixins/_card.scss`

This file defines the `card`, `panel-header`, `section`, and
`section-card` mixins. The components file
`scss/components/_cards.scss` just re-exposes `card` and `section-card`
on the default class selectors — DO NOT modify the components file.

Open `c:\laragon\www\ln-acme\scss\config\mixins\_card.scss` and apply
the changes below. Each change has a one-line justification.

### 3.1 — `@mixin panel-header` — header padding

```scss
// before
@mixin panel-header {
	@include flex;
	@include items-center;
	@include justify-between;
	@include px(1rem);
	@include py(0.75rem);
	@include border-b;
	@include bg-secondary;
```

```scss
// after
@mixin panel-header {
	@include flex;
	@include items-center;
	@include justify-between;
	@include px(var(--density-pad-md));
	@include py(var(--density-pad-sm));
	@include border-b;
	@include bg-secondary;
```

**Why:** panel header padding is content-area chrome — shrinks with
density so a card with header takes proportionally less vertical
space. `px` becomes `--density-pad-md` (1rem→0.625rem) and `py`
becomes `--density-pad-sm` (0.5rem→0.375rem). Note that the original
`py(0.75rem)` did not match any `--spacing-*` value exactly; mapping
to `--density-pad-sm` produces 0.5rem in comfortable (slightly less
than the original 0.75rem). This is an acceptable normalization to
the density scale — tokens win over one-off values. The visual delta
is tiny (4px) and consistent with the rest of the new system.

The inner `h3` font (`@include text-base; @include font-semibold`)
is **not changed** — it is a structural label, not body text.

### 3.2 — `@mixin section` header — `mb`, `pb` stay literal

In `@mixin section`, the inner `header { @include mb(1rem); @include pb(0.75rem); }`
is **NOT** density-aware. `section` is a top-of-page layout primitive
(used for `<section>` elements as full-width content groupings), not
a content-density component. Its rhythm is a layout decision, not an
information-density decision.

**Leave `@mixin section` unchanged.** Justification: section is a
layout rhythm primitive that nests density-aware components inside
itself; if its own rhythm shrank too, headings would crowd against
content unpredictably across pages.

### 3.3 — `@mixin section-card` — main and footer padding

```scss
// before
@mixin section-card {
	@include bg-primary;
	@include border;
	@include rounded-md;
	@include overflow-hidden;
	box-shadow: var(--shadow-xs);
	@include mb(1rem);

	> header {
		@include panel-header;
	}

	> main {
		@include p(1rem);

		&:has(> table),
		&:has(> .table-container) {
			@include p(0);

			> table,
			> .table-container > table {
				border-radius: 0;
				box-shadow: none;
			}
		}
	}

	> footer {
		@include px(1rem);
		@include py(0.75rem);
		@include border-t;
		@include flex;
		@include justify-end;
		@include gap(0.75rem);
	}
}
```

```scss
// after
@mixin section-card {
	@include bg-primary;
	@include border;
	@include rounded-md;
	@include overflow-hidden;
	box-shadow: var(--shadow-xs);
	@include mb(1rem);

	> header {
		@include panel-header;
	}

	> main {
		@include p(var(--density-pad-md));

		&:has(> table),
		&:has(> .table-container) {
			@include p(0);

			> table,
			> .table-container > table {
				border-radius: 0;
				box-shadow: none;
			}
		}
	}

	> footer {
		@include px(var(--density-pad-md));
		@include py(var(--density-pad-sm));
		@include border-t;
		@include flex;
		@include justify-end;
		@include gap(var(--density-gap-md));
	}
}
```

**Per-line justification:**

| Line | Before | After | Why |
|---|---|---|---|
| `> main { @include p(1rem); }` | `1rem` | `var(--density-pad-md)` | Card body padding is the canonical density target — this is the rule the plan's example shows. |
| `> main { ... @include p(0); }` (the `:has` clause) | `0` | `0` | Stay literal. Zero is structural — when a table or table-container is the direct child, the table's own borders take over. Density should not introduce padding here. |
| `> footer { @include px(1rem); }` | `1rem` | `var(--density-pad-md)` | Footer is content-area chrome, mirrors header. |
| `> footer { @include py(0.75rem); }` | `0.75rem` | `var(--density-pad-sm)` | Same logic as panel-header py. |
| `> footer { @include gap(0.75rem); }` | `0.75rem` | `var(--density-gap-md)` | Gap between footer action buttons can shrink in compact mode — buttons themselves stay full size, but they sit closer together. |
| `@include mb(1rem);` (between cards) | `1rem` | `1rem` (unchanged) | Margin between cards is a page-rhythm decision, not a within-card density decision. Leave literal. |

The `@mixin card` itself (the bare card, no header/main/footer
structure) does NOT have any padding in its body — it relies on
inner `> header`, `> main`, `> footer` rules from `section-card`.
**No change needed in `@mixin card`.**

### 3.4 — Card accents — leave unchanged

`@mixin card-accent-top`, `card-accent-bottom`, `card-accent-left`,
`card-bg`, `card-stacked` use `border-top: 3px`, `bottom: -5px`, etc.
These are decorative pixel values, not content padding.
**Leave all card-accent mixins unchanged.**

`form-grid` uses `@include gap(var(--spacing-md));` for the grid gap

**Leave unchanged.** The form grid gap is a 6-column layout decision.

	@include p(0);
	@include flex;
	@include justify-end;
	@include gap(0.75rem);
	@include mt(1.5rem);
	@include pt(1rem);
	@include border-t;
}
```

**Leave entirely unchanged.** Form actions row is structurally
similar to a page footer — its rhythm separates the form from its

here (there is no information; only two buttons). Buttons themselves
do not shrink (a11y), so shrinking the gap and margin around them

### 4.4 — `@mixin form-input` — the canonical density target

```scss
// before
@mixin form-input {
	@include w-full;
	@include px(0.875rem);
	@include py(0.5rem);
	@include text-base;
	@include text-primary;
	@include bg-primary;
	@include border;
	@include rounded-md;
	@include transition;
	outline: none;
```

```scss
// after
@mixin form-input {
	@include w-full;
	@include px(var(--density-pad-md));
	@include py(var(--density-pad-sm));
	@include text-base;
	@include text-primary;
	@include bg-primary;
	@include border;
	@include rounded-md;
	@include transition;
	outline: none;
```

**Per-line justification:**

| Line | Before | After | Why |
|---|---|---|---|
| `@include px(0.875rem);` | `0.875rem` | `var(--density-pad-md)` | Horizontal input padding is the dominant visible density — comfortable 1rem feels generous, compact 0.625rem feels dense. The minor delta from 0.875→1rem in comfortable mode is acceptable normalization. |
| `@include py(0.5rem);` | `0.5rem` | `var(--density-pad-sm)` | Vertical input padding determines input height. 0.5rem comfortable matches the original; 0.375rem compact is the dense floor. |
| `@include text-base;` | unchanged | unchanged | **Do NOT shrink input text size by density.** Input text is user content. Forcing it to shrink with density makes typing feel like reading fine print. The user's content should stay legible regardless of density mode. |

**Question: should we set `min-height: var(--density-row-h)` on the input explicitly?**

**Answer: NO.** Padding alone determines visible input height
(0.5rem × 2 + ~1.5rem line-height = ~2.5rem comfortable;
0.375rem × 2 + ~1.5rem = ~2.25rem compact). Adding an explicit
`min-height` creates two competing height sources, and any custom
input variant a consumer adds (e.g. with an icon prefix) would have
to fight the min-height. Let padding determine height naturally.
The `--density-row-h` token exists for **table rows** where padding
is on `td` and the row height is implicit — there it acts as a
floor for short content (see Step 5).

### 4.5 — `@mixin form-textarea` — `min-height` is an editor concern

`form-textarea { min-height: 6rem; resize: vertical; }`.

**Leave unchanged.** 6rem is the minimum sensible textarea size —
shrinking to ~5rem in compact mode does not improve density meaningfully
and harms multi-line text editing.

### 4.6 — `@mixin form-select` — arrow position is structural

`form-select` uses `padding-right: 2.5rem;` to make room for the
arrow icon, and `background-position: right 0.75rem center;` to
position the arrow.

**Leave unchanged.** These values are tied to the SVG arrow's size
(12×12px) and cannot shrink without the arrow visually colliding with
the text. The horizontal `px` from `form-input` already shrinks the
left side appropriately.

### 4.7 — `@mixin form-check`, `form-checkbox`, `form-radio`

`form-check` uses `@include size(1.25rem);` (20×20 hit target).

**Leave unchanged.** Checkboxes and radios are interactive controls —
shrinking them below 20px hurts accessibility (touch hit target).
Same reasoning as buttons.

### 4.8 — `@mixin pill`, `pill-group`

`pill` uses `@include px(1rem); @include py(0.5rem);` — these are
button-like interactive elements.

**Leave unchanged.** Pills are tap targets, not content. Same a11y
reasoning as buttons.

**Acceptance criterion for Step 4:** the only edit in `_form.scss`
is inside `@mixin form-input`: two lines (`px` and `py`) reference
`var(--density-pad-md)` and `var(--density-pad-sm)`. All other

### 5.3 — `@mixin table-base` — `tbody tr` row height floor

Add a `min-height` rule to `tbody tr`. Currently:

```scss
// before
tbody tr {
	@include transition;

	&:hover {
		@include bg-secondary;
	}
}
```

```scss
// after
tbody tr {
	@include transition;
	min-height: var(--density-row-h);

	&:hover {
		@include bg-secondary;
	}
}
```

**Why:** padding alone gives the row a natural height proportional
to text size. For very short cell content (e.g. a single icon, or
a checkbox), the row collapses below the comfortable scanline. The
`min-height` floor (2.75rem comfortable, 2.25rem compact) keeps every
row scannable. **Note:** `min-height` on `<tr>` works inconsistently
across browsers — apply it as a hint, not a guarantee. The `td`
padding is still doing the real work; this is a safety net for
sparse content.

If you find during build/test that `min-height: <value>` on `tr` is
silently ignored by the layout engine and rows still collapse, add
the same `min-height` to `td` instead and remove it from `tr`. Note
this in your final report. (Either location is acceptable — the goal
is "rows have a floor.")

### 5.4 — `@mixin table-responsive` — leave alone

`table-responsive` is the mobile-stacked variant (cards-from-rows
pattern). Its `@include px(1rem); @include py(0.75rem);` are already
content padding for the stacked layout. Density is meant for desktop
data-density use cases; on mobile the layout is already adapted via
the container query (the table-container's `@container table (max-width: 640px)`).

**Leave `table-responsive` unchanged.** Density × responsive-stacked
is an interaction nobody asked for, and mixing the two would create
test-matrix explosion for negligible benefit.

### 5.5 — `@mixin table-striped`, `table-section-header`, `table-action`

**Leave all three unchanged.**

- `table-striped` only changes background — no spacing.

## Step 7 — Create `docs/css/density.md`

Create a new file at `c:\laragon\www\ln-acme\docs\css\density.md`
with these contents:

```markdown
# Density

File: `scss/config/_density.scss`. Added in v1.2.

## Philosophy

ln-acme v1.2 supports a global density toggle for data-dense UIs.
Compliance dashboards, audit log tables, settings pages with many
fields — these benefit from showing more rows on screen. Marketing
pages, modals, and onboarding screens do not.

Density is implemented as a **CSS variable override**, not a
component override. A single attribute (`data-density="compact"`)
re-tunes the `--density-*` token scale, and every component
referencing those tokens shrinks automatically.

## Activation

Three activation paths:

### 1. Global

```html
<html data-density="compact">
```

The whole document switches to compact spacing. Persist user choice
client-side (`localStorage`, cookie); the library does not persist.

### 2. Scoped

```html
<main data-density="compact">
	<!-- only this region is compact -->
</main>
```

Density is a normal CSS custom property cascade — it works at any
DOM scope, not just `<html>`. Use this for "compact data table on a
comfortable page" layouts.

### 3. Default (comfortable)

No attribute = comfortable mode. There is no `data-density="comfortable"`
form — comfortable IS the default `:root` state. (You may write the
attribute explicitly to UN-set a parent `compact`, e.g. embedding a
comfortable preview inside a compact page; the override re-defines
the tokens back to their defaults.)

## Token reference

All tokens defined in `scss/config/_density.scss`.

### Padding scale

| Token | Comfortable | Compact | Used by |
|---|---|---|---|
| `--density-pad-xs` | `0.25rem` | `0.125rem` | reserved — not currently consumed |
| `--density-pad-sm` | `0.5rem` | `0.375rem` | input `py`, table cell `py`, panel header `py`, section-card footer `py` |
| `--density-pad-md` | `1rem` | `0.625rem` | input `px`, table cell `px`, panel header `px`, card body `p`, section-card footer `px` |
| `--density-pad-lg` | `1.5rem` | `1rem` | stat-card `p` |

### Gap scale

| Token | Comfortable | Compact | Used by |
|---|---|---|---|
| `--density-gap-sm` | `0.5rem` | `0.375rem` | reserved |
| `--density-gap-md` | `0.75rem` | `0.5rem` | section-card footer button gap |
| `--density-gap-lg` | `1rem` | `0.75rem` | reserved |

### Row height

| Token | Comfortable | Compact | Used by |
|---|---|---|---|
| `--density-row-h` | `2.75rem` | `2.25rem` | table `tbody tr` min-height (floor for sparse rows) |
| `--density-row-h-sm` | `2.25rem` | `1.875rem` | reserved |

### Body typography

| Token | Comfortable | Compact | Used by |
|---|---|---|---|
| `--density-font-body` | `var(--text-body-md)` (1rem) | `var(--text-body-sm)` (0.875rem) | table `td` font-size |
| `--density-lh-body` | `var(--lh-body-md)` (1.6) | `var(--lh-body-sm)` (1.5) | table `td` line-height |

## What reacts to density

| Component | Reacts? | What changes |
|---|---|---|
| Tables | YES | header cell padding, body cell padding, body cell font + line-height, row min-height |
| Forms (inputs) | YES | input/textarea/select horizontal and vertical padding |
| Cards (`section-card`) | YES | body padding, footer padding, footer button gap |
| Panel header (cards, dialogs) | YES | header padding |
| Stat card | YES | outer padding |

## What does NOT react to density and why

| Component | Why not |
|---|---|
| Buttons | Hit-target accessibility — WCAG 2.5.5 recommends ≥44px. Shrinking buttons by density violates this floor. |
| Form labels | Structural — labels NAME fields and must stay legible at any density. |
| Form actions row (`.form-actions`) | Contains buttons; cramping the gap doesn't save space if buttons themselves are full size. |
| Form grid gap | Layout — adjacent fields' focus rings must not overlap. |
| Pills, checkboxes, radios | Interactive controls; same a11y reasoning as buttons. |
| Modals | A modal is a focused interruption with its own scale. Shrinking modal padding makes it feel cramped without showing more data on screen. |
| Toast | Ephemeral, fixed visual size. Not a data display surface. |
| Page header | Fixed visual hierarchy at the top of the page. Reducing it does not improve information density of the content area. |
| Sidebar nav | Density semantics for nav are ambiguous (does compact nav mean smaller fonts, tighter rows, no labels?). Deferred until a consumer asks. |
| Card accents (`card-accent-top` etc.) | Decorative pixel borders, not content padding. |
| Section margin (`section { mb: 2rem }`) | Page rhythm primitive, not within-card density. |
| Stat card label / value / trend typography | The big number IS the stat card's identity; shrinking it defeats the component's purpose. |
| Table headers (`th` font) | Column labels are structural — must stay scannable. Header padding shrinks; header font does not. |
| Mobile-stacked table (`table-responsive`) | Already adapted by container query for small viewports; layering density would create test-matrix explosion. |

## Use cases

- **Audit log table** — hundreds of rows; user wants to see as many
  as possible. Activate compact globally.
- **Settings page with many fields** — flip compact for power users
  who navigate by keyboard and prefer to see all fields without
  scrolling.
- **Embedded data table inside a comfortable dashboard** — scope
  compact to the table only.
- **User preference toggle** — surface a setting "Compact density"
  in the user's profile; persist with `localStorage`.

## Anti-patterns

- **Auto-switching density by viewport.** Density is an explicit
  user choice. Viewport breakpoints handle layout (sidebar shows/hides),
  density handles information density inside that layout. Mixing the
  two creates surprise: the user resizes a window and content
  unexpectedly shrinks.
- **Per-component density selectors.** Do NOT write
  `[data-density="compact"] [data-ln-card] { padding: ... }`.
  That explodes specificity and duplicates rules. Use the
  `--density-*` tokens — components reference them, the override
  flows through the cascade.
- **Shrinking buttons or interactive controls.** WCAG hit target
  size is non-negotiable. Density shrinks chrome and content, never
  controls.
- **Shrinking body content text in non-table components.** Tables are
  the one exception (column scanability); everywhere else, content
  text stays at its declared size.
- **Defining new `--density-*` tokens per component.** The token
  surface is intentionally small (4 padding steps + 3 gap steps + 2
  row heights + body font/lh). New components opt in by referencing
  existing tokens, not by adding new ones.

## Dark mode interaction

Density and dark mode are orthogonal axes. Density only touches
`--density-*` tokens; dark mode (`scss/config/_theme.scss`) only
touches `--color-*` tokens. They cannot collide, and they compose:

```html
<html data-theme="dark" data-density="compact">
```

A compact, dark-mode UI works exactly as expected — every component
is both dark and dense.

## Container query interaction

Density only changes a component's INTERNAL spacing. It does NOT
change a component's outer dimensions. A filter-toolbar with a
container query at 880px still flips at 880px in both comfortable
and compact modes — the density change does not push the toolbar
across its breakpoint.

## Adding a new component to density

1. Identify spacing rules in the component's mixin file.
2. Decide for each: structural (border, label, control hit target,
   layout rhythm) or content (cell padding, card body, panel header)?
3. Replace content rules with `var(--density-pad-*)` /
   `var(--density-gap-*)`.
4. Leave structural rules literal.
5. Document the component in this file's "What reacts" table.
```

**Acceptance criterion:** file exists, is valid markdown, every
internal reference (component names, token names) matches the
actual files.
