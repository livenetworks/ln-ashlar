## Scope

Three additions. Each has a concrete "when would a consumer use this"
answer — no speculative components.

| # | Component | Type | Purpose |
|---|---|---|---|
| A | `ln-popover` | JS | Click-triggered rich-content panel (richer than dropdown, lighter than modal) |
| B | Density system | SCSS | `[data-density="compact"]` attribute — global spacing shrink for data-dense UIs |
| C | `ln-tooltip` enhance | JS | Smart positioning + `aria-describedby` layer over the Phase 7d CSS tooltip |

## Explicit non-goals (previously considered, dropped)

- **`ln-copy`** — 10 lines inline JS solves it. No reusable value.
- **`ln-shortcuts`** — compliance-officer users don't operate in
  power-user keyboard mode. No concrete consumer need.
- **ln-popover form helpers** — popover is a primitive, not a forms
  library. Consumer composes form markup inside the popover slot.
- **Density breakpoint auto-toggle** — density is explicit user choice
  (settings toggle), never auto-switched by viewport.

## Shared infrastructure

Both popover and tooltip enhance need the same positioning and portal
logic. Ship it once in ln-core, not twice in the components.

### `js/ln-core/positioning.js`

Exports:

```javascript
// Compute best placement for a floating element relative to an anchor.
// Returns { top, left, placement } in viewport pixels.
//
// preferred: 'top' | 'bottom' | 'right' | 'left'
// fallback order: preferred → opposite → perpendicular pair
// If none fit fully, pins to viewport edge.
export function computePlacement(anchorRect, floatingSize, preferred, offset = 4)

// Move an element into <body>, remember origin, return a restore fn.
// Use for portals (popovers, tooltips, toasts).
export function teleportToBody(el)

// Measure an element that may be hidden (temporarily makes it visible
// off-screen). Returns { width, height }.
export function measureHidden(el)
```

**Design notes:**

- Pure functions where possible (`computePlacement` takes a rect, not
  a DOM node — easy to test, no side effects).
- `teleportToBody` returns a cleanup callback so the component doesn't
  track state in its own instance.
- `measureHidden` is a helper for the common pattern: "I need to know
  how big this popover is before I position it, but showing it first
  causes a flicker."
- **Do NOT touch `ln-dropdown`.** It has its own internal `_positionMenu`
  and `_teleportToBody`. Unifying is out of scope — works, not broken,
  refactor risk without matching benefit.

### `js/ln-core/index.js`

Re-export the three new functions alongside existing helpers.

## Scope A — ln-popover

### What it is

Click-triggered container for rich content. Answers the "dropdown is
semantically wrong for a user profile menu with mixed content, but
modal is too heavy" gap.

Use cases in the intended consumers:
- Header avatar click → panel with "Signed in as X", "Settings", "Sign out"
- `i` icon next to a field → explanation with learn-more link
- Status badge click → "Changed by X on date Y, reason Z"

### Architecture

**Markup:**

```html
<button type="button" data-ln-popover-for="user-menu">
	<svg class="ln-icon" aria-hidden="true"><use href="#ln-user"></use></svg>
</button>

<div data-ln-popover="closed" id="user-menu">
	<h4>Account</h4>
	<p>Signed in as <strong>admin@example.com</strong></p>
	<hr>
	<button type="button">Settings</button>
	<button type="button">Sign out</button>
</div>
```

**Rules:**

- `data-ln-popover-for="<id>"` on trigger — points to target by id
- `data-ln-popover="open|closed"` on target — state attribute driven
- State is a string attribute, not a boolean, for consistency with
  ln-modal (`data-ln-modal="open"`)
- MutationObserver on target watches `data-ln-popover` — mutations
  from outside (e.g., another script calling `el.setAttribute`) also
  trigger show/hide
- Click on trigger → set `data-ln-popover="open"`
- Click outside the popover AND outside the trigger → close all open
- `Escape` key → close all open (listener added only while any open,
  removed when none)
- Popover receives focus on open (first focusable element, or the
  popover itself if none); focus returns to trigger on close
- Positioning: default `bottom`, falls back via `computePlacement`

**Co-located SCSS** (`js/ln-popover/ln-popover.scss`):

```scss
@use '../../scss/config/mixins' as *;

[data-ln-popover] {
	display: none;
	position: fixed;
	min-width: 14rem;
	max-width: 24rem;
	@include p(1rem);
	background: hsl(var(--color-bg-primary));
	border: 1px solid hsl(var(--color-border));
	@include rounded-lg;
	box-shadow: var(--shadow-lg);
	z-index: var(--z-dropdown);

	&[data-ln-popover="open"] {
		display: block;

		@include motion-safe {
			animation: ln-popover-fade var(--transition-fast);
		}
	}
}

@keyframes ln-popover-fade {
	from { opacity: 0; transform: translateY(-4px); }
	to   { opacity: 1; transform: translateY(0); }
}
```

## Scope B — Density system

### What it is

A global spacing compression toggle. Flip one attribute, entire UI
tightens. Tables show more rows, forms pack denser, cards shrink
padding. Same components, same markup, just less whitespace.

Consumer activates:
```html
<html data-density="compact">
```

Or scoped:
```html
<main data-density="compact">
	<!-- everything here is compact -->
</main>
```

### Architecture

**CSS variable override pattern** — not a component override.

Density introduces a set of `--density-*` tokens that default to
"comfortable" values and get re-tuned under `[data-density="compact"]`:

```scss
// scss/config/_density.scss
:root {
	--density-pad-xs:  0.25rem;
	--density-pad-sm:  0.5rem;
	--density-pad-md:  1rem;
	--density-pad-lg:  1.5rem;

	--density-gap-sm:  0.5rem;
	--density-gap-md:  0.75rem;
	--density-gap-lg:  1rem;

	--density-row-h:   2.75rem;   // table row, input height
	--density-row-h-sm: 2.25rem;

	--density-font-body: var(--text-body-md);
	--density-lh-body:   var(--lh-body-md);
}

[data-density="compact"] {
	--density-pad-xs:  0.125rem;
	--density-pad-sm:  0.375rem;
	--density-pad-md:  0.625rem;
	--density-pad-lg:  1rem;

	--density-gap-sm:  0.375rem;
	--density-gap-md:  0.5rem;
	--density-gap-lg:  0.75rem;

	--density-row-h:   2.25rem;
	--density-row-h-sm: 1.875rem;

	--density-font-body: var(--text-body-sm);
	--density-lh-body:   var(--lh-body-sm);
}
```

**Components adopt density-aware tokens selectively.** Not every
component needs to react — only the ones where users actually want
more data on screen:

| Component | Density-aware? | Which tokens |
|---|---|---|
| Tables | YES | row height, cell padding, font size |
| Forms (inputs) | YES | input height, label spacing |
| Cards | YES | padding |
| Stat card | YES | padding, value size |
| Page header | NO | fixed visual hierarchy |
| Modal | NO | modal is already its own scale |
| Toast | NO | ephemeral, fixed size |
| Buttons | NO | hit target size is accessibility concern — don't shrink below 44px |
| Sidebar nav | Maybe | density is ambiguous for nav — defer unless consumer asks |

### Retrofit strategy

Components currently hard-code spacing via `@include p(1rem)`. The
density retrofit replaces those with density tokens in the components
that opted in:

```scss
// scss/components/_cards.scss — before
[data-ln-card] {
	@include p(1rem);
}

// after
[data-ln-card] {
	padding: var(--density-pad-md);
}
```

**Anti-pattern to avoid:** do NOT introduce a separate
`[data-density="compact"] [data-ln-card] { ... }` selector. That
explodes specificity and duplicates rules. Use tokens.

|---|---|

| `scss/ln-acme.scss` | `@use` the new config file (after `_theme`) |

## Scope C — ln-tooltip enhance

### What it is

Progressive enhancement over the Phase 7d CSS baseline
(`[data-ln-tooltip]`). CSS baseline works for 80% of cases. JS enhance
solves two specific problems:

1. **Viewport clipping** — when tooltip would render off-screen, JS
   flips it to a side that fits.
2. **Proper `aria-describedby`** — CSS can't wire up accessible
   descriptions. JS creates a real `<div>` in a portal and points
   `aria-describedby` at it.

CSS and JS coexist. Opt-in to JS per element:

```html
<!-- CSS-only (Phase 7d) — unchanged behavior -->
<button data-ln-tooltip="Edit document">…</button>

<!-- JS-enhanced — flips to fit, wires aria-describedby -->
<button data-ln-tooltip="Edit document" data-ln-tooltip-enhance>…</button>
```

### Architecture

- `DOM_SELECTOR = '[data-ln-tooltip-enhance]'`
- On mount: walk all matching, assign each a stable uid
- Portal container `<div id="ln-tooltip-portal">` created on first use
  (singleton, appended to `<body>`)
- `mouseenter` / `focus`:
  - Create `<div class="ln-tooltip">` inside the portal
  - Fill text from `data-ln-tooltip`
  - `computePlacement(trigger.getBoundingClientRect(), { w, h }, data-ln-tooltip-position || 'top')`
  - Set `aria-describedby` on the trigger pointing at the new node
- `mouseleave` / `blur`:
  - Remove the tooltip node
  - Clear `aria-describedby`
- MutationObserver for dynamically added triggers
- ESC while tooltip is visible → hide (same as `blur`)

**Co-located SCSS** (`js/ln-tooltip/ln-tooltip.scss`):

```scss
@use '../../scss/config/mixins' as *;

#ln-tooltip-portal {
	position: fixed;
	top: 0;
	left: 0;
	pointer-events: none;
	z-index: var(--z-toast);
}

.ln-tooltip {
	position: absolute;
	@include px(0.5rem);
	@include py(0.25rem);
	background: hsl(var(--color-neutral-900));
	color: hsl(var(--color-neutral-50));
	@include typography(caption);
	@include rounded-sm;
	white-space: nowrap;
	box-shadow: var(--shadow-md);
	max-width: 20rem;

	@include motion-safe {
		animation: ln-tooltip-fade var(--transition-fast);
	}
}

@keyframes ln-tooltip-fade {
	from { opacity: 0; transform: translateY(2px); }
	to   { opacity: 1; transform: translateY(0); }
}
```

**Does NOT replace the CSS baseline.** Both coexist — CSS runs on
`[data-ln-tooltip]:not([data-ln-tooltip-enhance])`, JS runs on
`[data-ln-tooltip-enhance]`. The CSS in `scss/components/_tooltip.scss`
already uses just `[data-ln-tooltip]` so consumers with both attributes
get both behaviors — JS hides the CSS one by clearing the
pseudo-element content on enhance. Verify this during implementation.

   Depends on commit 1. Touches co-existing CSS baseline — verify
   CSS continues to work on un-enhanced tooltips.

5. **`build: rebuild dist for v1.2 scope`**
   Standard trailing build commit. Never committed with feature code.

Merge to main with `--no-ff` after verification.
