# Plan: v1.1 Phase 8 — JS-Enhanced Components

**Source roadmap:** `plans/design-system-v1.1-roadmap.md` § Phase 8
**Branch:** `v1.1/phase-8-js-components`
**Depends on:** Phase 7d (CSS tooltip baseline, chip)

**Components in this batch:**
1. `ln-tooltip` — JS enhance for smart positioning + `aria-describedby`
2. `ln-popover` — click-triggered rich content slot
3. `ln-copy` — copy-to-clipboard with fade confirmation
4. `ln-shortcuts` — global keyboard shortcut registry with `?` overlay

---

## Context

Phase 7d shipped a CSS-only tooltip via `data-ln-tooltip`. That works
for 80% of cases but breaks when the tooltip would render off-screen
(viewport clipping) and cannot host interactive content. Phase 8
adds the JS layer that solves those cases.

The other three JS components are small, high-value additions:
`ln-popover` for rich dropdowns that aren't form selects, `ln-copy`
because every admin surface has a "copy ID" button, and `ln-shortcuts`
because keyboard-heavy power users need a registry to add/remove
shortcuts globally.

Each component follows `CLAUDE.md` § "Adding a New JS Component":
IIFE pattern, `data-ln-*` attribute, `<template>` for DOM structure,
`js/ln-{name}/README.md`, `docs/js/{name}.md`, `demo/admin/{name}.html`.

## Pre-resolved ambiguities

### A. ln-tooltip — architecture

Enhances the existing CSS baseline. On mount:

1. Finds all `[data-ln-tooltip]` elements
2. For each, generates an `id` and wires `aria-describedby`
3. On `mouseenter` / `focus`, creates a floating `<div>` inside a
   portal container (`ln-acme-tooltip-portal` appended to `<body>`)
4. Positions with viewport-collision logic: try `data-ln-tooltip-position`
   first, fall back through `top → bottom → right → left`
5. Removes on `mouseleave` / `blur`

**Does not replace the CSS baseline.** Both coexist — CSS tooltip
runs on elements without `data-ln-tooltip-enhance`, JS tooltip runs
on those with the enhance attribute.

### B. ln-popover — architecture

Click-triggered. Reference existing `ln-dropdown` for positioning
helpers. Key differences from ln-dropdown:
- Rich content slot (not a list of options)
- Can contain forms, buttons, anything
- Dismisses on outside-click OR Escape

```html
<button type="button" data-ln-popover-for="user-info">
	<svg class="ln-icon" aria-hidden="true"><use href="#ln-info-circle"></use></svg>
</button>

<div data-ln-popover id="user-info">
	<h4>Account</h4>
	<p>Signed in as admin@example.com</p>
	<button type="button">Sign out</button>
</div>
```

State stored on `data-ln-popover` attribute (`open`/`closed`), same
pattern as `ln-modal`.

### C. ln-copy

Zero-config:

```html
<button type="button" data-ln-copy="Document ID: 12345">
	Copy ID
</button>
```

On click: `navigator.clipboard.writeText()`, then flash a confirmation
message next to the button for 2 seconds, then fade out. No new DOM
elements — the confirmation is a `::after` pseudo-element driven by a
`data-ln-copy-state="copied"` attribute applied for 2s.

### D. ln-shortcuts

Global registry. Zero HTML markup required on the app side beyond
imports. Public API:

```javascript
import { registerShortcut, unregisterShortcut, showShortcutHelp } from 'ln-acme';

registerShortcut('Ctrl+K', () => openCommandPalette(), 'Open command palette');
registerShortcut('?', showShortcutHelp, 'Show shortcut help');
registerShortcut('g d', () => navigate('/dashboard'), 'Go to dashboard'); // sequence
```

`showShortcutHelp()` opens an overlay listing all registered
shortcuts grouped by scope. Overlay is a self-contained
`<dialog>` or absolute-positioned panel.

One commit per component:

### Step 2 — ln-tooltip (Commit 1)

Create `js/ln-tooltip/ln-tooltip.js`:

- IIFE shell matching `ln-modal.js` structure
- `DOM_SELECTOR = '[data-ln-tooltip-enhance]'`
- On mount: walk all matching elements, assign unique IDs
- Create portal container `<div id="ln-tooltip-portal">` once on first use
- `mouseenter` / `focus` handler:
  - Create `<div class="ln-tooltip">` in portal
  - Read text from `data-ln-tooltip` attribute
  - Compute position per `data-ln-tooltip-position` with fallback:
    1. Try requested position
    2. If off-screen, rotate through top → bottom → right → left
    3. If none fit, pin to viewport edge
  - Set `aria-describedby` on the trigger pointing to the new tooltip
- `mouseleave` / `blur` handler:
  - Remove tooltip node
  - Clear `aria-describedby`
- MutationObserver for dynamic elements

Co-located SCSS `js/ln-tooltip/ln-tooltip.scss`:

```scss
@use '../../scss/config/mixins' as *;

#ln-tooltip-portal {
	position: fixed;
	top: 0;
	left: 0;
	pointer-events: none;
	z-index: var(--z-toast);
}

.ln-tooltip {
	position: absolute;
	@include px(0.5rem);
	@include py(0.25rem);
	background: hsl(var(--color-neutral-900));
	color: hsl(var(--color-neutral-50));
	@include typography(caption);
	@include rounded-sm;
	white-space: nowrap;
	box-shadow: var(--shadow-md);

	@include motion-safe {
		animation: ln-tooltip-fade var(--transition-fast);
	}
}

@keyframes ln-tooltip-fade {
	from { opacity: 0; transform: translateY(2px); }
	to   { opacity: 1; transform: translateY(0); }
}
```

Create `js/ln-tooltip/README.md`, `docs/js/tooltip.md`, `demo/admin/ln-tooltip.html`.

### Step 3 — ln-popover (Commit 2)

`js/ln-popover/ln-popover.js`:

- `DOM_SELECTOR = '[data-ln-popover]'`
- Trigger attribute: `data-ln-popover-for="id"`
- State attribute on target: `data-ln-popover="open|closed"`
- MutationObserver watches the state attribute to handle show/hide
- Click on trigger: open the target (set attribute)
- Click outside: close all open popovers
- Escape key: close all open popovers (listener only while any open)
- Positioning: same fallback logic as `ln-tooltip`, but richer
  (popovers are wider, need more space)

Co-located SCSS with animation gated in `motion-safe`.

Files: `js/ln-popover/ln-popover.js`, `ln-popover.scss`, `README.md`,

### Step 4 — ln-copy (Commit 3)

`js/ln-copy/ln-copy.js`:

- `DOM_SELECTOR = '[data-ln-copy]'`
- Click handler:
  - Read text from attribute value
  - `navigator.clipboard.writeText(text)`
  - Set `data-ln-copy-state="copied"` on element
  - After 2s, remove the state attribute
- MutationObserver for dynamic elements

Co-located SCSS:

```scss

[data-ln-copy] {
	position: relative;

	&[data-ln-copy-state="copied"]::after {
		content: 'Copied';
		position: absolute;
		top: -2rem;
		left: 50%;
```

### Step 5 — ln-shortcuts (Commit 4)

`js/ln-shortcuts/ln-shortcuts.js`:

- No DOM selector (programmatic API only)
- Internal registry: `Map<shortcut, { handler, label, scope }>`
- Public API:
  - `registerShortcut(combo, handler, label?)` — combo is a string like `'Ctrl+K'` or `'g d'` (sequence)
  - `unregisterShortcut(combo)`
  - `showShortcutHelp()` — opens overlay
  - `hideShortcutHelp()`
- Global keydown listener:
  - Parse `event.key`, modifiers (`ctrlKey`, `metaKey`, `shiftKey`, `altKey`)
  - Normalize to canonical combo string
  - Look up in registry, call handler
  - Sequence support: track pending keys with a 1000ms timeout
- Help overlay: `<dialog>` element with a list of registered shortcuts,
  keybinding shown as `<kbd>` (Phase 7c)

Co-located SCSS for the help overlay.

Export public API from `js/index.js`.

Files: `ln-shortcuts.js`, `ln-shortcuts.scss`, `README.md`,
`docs/js/shortcuts.md`, `demo/admin/ln-shortcuts.html`.
